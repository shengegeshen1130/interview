### Q1 (10:00) - Continuous Batching / Dynamic Batching（推理服务调度核心）
**题目：** 什么是Static Batching和Continuous Batching（也叫Iteration-level Batching）？为什么LLM推理中传统Static Batching效率极低？Continuous Batching如何解决"短请求等长请求"的问题？vLLM/TGI中的具体实现机制？Prefill和Decode阶段的调度冲突（Chunked Prefill）？吞吐量 vs 延迟的trade-off？

**参考答案：**

1. **Static Batching的问题：**
   - 传统方式：收集B个请求，组成一个batch，等所有请求都生成完毕才释放
   - 问题：LLM生成长度差异巨大（有的生成10 tokens，有的生成500 tokens）
   - 短请求生成完毕后只能padding等待→GPU空转→利用率极低
   - 类比：餐厅要求所有人吃完才能离开→吃得快的人浪费时间

2. **Continuous Batching核心思想（Orca, Yu et al. 2022）：**
   - 在每个iteration（每生成一个token）级别进行调度
   - 某个请求生成完毕（遇到EOS）→立即从batch中移除→新请求立即加入
   - GPU始终满负载：不存在"等别人"的情况
   - 效果：吞吐量提升2-8x（相比static batching），尤其在请求长度方差大时

3. **具体实现机制（vLLM为例）：**
   - **Scheduler调度器：** 每个iteration检查：
     ① 哪些请求完成了→移出
     ② 等待队列中有新请求→加入（如果KV Cache内存允许）
     ③ 如果内存不够→对优先级低的请求做preemption（swap到CPU或recompute）
   - **PagedAttention配合：** KV Cache按页分配→请求完成后释放页→新请求立即使用→无碎片
   - **Preemption策略：** 内存不足时，swap（KV Cache写到CPU内存，恢复时读回）或recompute（丢弃KV Cache，重新prefill）

4. **Prefill vs Decode调度冲突：**
   - **问题：** Prefill是计算密集（大量矩阵乘），Decode是内存密集（逐token生成，bandwidth-bound）
   - 混在同一batch中：Prefill拖慢正在decode的请求（TTFT影响已有请求的TPOT）
   - **Chunked Prefill：** 将长prompt的prefill分成小块（如512 tokens一块），与decode交替执行
   - 效果：避免长prefill阻塞decode请求，降低已有请求的延迟抖动
   - **Sarathi-Serve / Splitwise：** 进一步将Prefill和Decode分到不同GPU，专门优化

5. **吞吐量 vs 延迟 Trade-off：**
   - batch size越大→吞吐量越高，但单个请求延迟增加
   - TTFT（Time To First Token）：受Prefill调度影响→排队等待
   - TPOT（Time Per Output Token）：受batch size影响→batch越大decode越慢
   - 生产策略：
     - 设置max_batch_size上限（如256）
     - 设置max_wait_time（如50ms）→不等凑满batch就发
     - 优先级队列：VIP请求优先调度
     - SLO-aware调度：根据延迟目标动态调整batch大小

6. **核心指标：**
   - **Throughput：** tokens/second（整体吞吐）
   - **TTFT：** 用户发送请求到收到第一个token的时间
   - **TPOT：** 每个output token的生成间隔
   - **Time-to-last-token（TTLT）：** 完整响应时间
   - 生产环境通常用P50/P99 TTFT和TPOT作为SLO

### Q2 (11:00) - KL散度实战应用串联（RLHF/蒸馏/VAE/DPO中的KL）（复习强化）
**题目：** KL散度在LLM训练中无处不在——请串联解释KL散度在以下四个场景中的具体作用和计算方式：①RLHF中约束Actor不偏离Reference Model ②知识蒸馏中Student学习Teacher ③VAE的ELBO中正则化潜变量 ④DPO中隐式的KL约束。四个场景中KL的方向（正向/反向）有什么讲究？KL散度的数值监控在训练中的实战意义？

**参考答案：**

1. **KL散度基础回顾：**
   - KL(P||Q) = Σ P(x) log(P(x)/Q(x))：从P的视角衡量Q的"信息损失"
   - 不对称：KL(P||Q) ≠ KL(Q||P)
   - 正向KL（KL(P||Q)）：P是目标，Q是近似→Q倾向覆盖P的所有模式（mode-covering）
   - 反向KL（KL(Q||P)）：Q是近似，P是目标→Q倾向锁定P的一个模式（mode-seeking）

2. **场景①：RLHF中的KL约束**
   - 目标：maximize R(y|x) - β·KL(π_θ || π_ref)
   - KL方向：KL(新策略 || 参考策略) = 反向KL（从新策略视角）
   - 作用：防止Actor为了追求高reward而"走太远"→reward hacking
   - 实例：不加KL约束→模型学会输出"I agree! Great question!"获得高reward但丧失多样性
   - β系数：通常0.01-0.1，太大→学不动，太小→reward hacking
   - 监控：训练中KL值持续上升说明模型在偏离→可能需要增大β或停止训练

3. **场景②：知识蒸馏中的KL**
   - 损失：KL(P_teacher || P_student) = 正向KL
   - KL方向：正向KL（Teacher是目标分布P）
   - 为什么用正向KL：Student需要覆盖Teacher的所有模式（不能遗漏知识）→mode-covering
   - 带温度的Softmax：P_i = exp(z_i/T) / Σexp(z_j/T)，T>1让分布更平滑→暗知识（dark knowledge）
   - 总损失 = α·KL_distill(T²缩放) + (1-α)·CE_hard_label
   - 实战：T=2-4最常用，α=0.5-0.9

4. **场景③：VAE的ELBO中的KL**
   - ELBO = E_q[log p(x|z)] - KL(q(z|x) || p(z))
   - KL方向：KL(后验近似 || 先验) = 反向KL
   - 作用：让编码器学到的潜变量分布接近标准正态N(0,I)→正则化
   - 没有KL项→潜空间不规则→无法生成/插值
   - KL坍塌问题：KL项优化过快→q(z|x)直接等于p(z)→z不编码任何信息→解码器忽略z
   - 解决：KL annealing（β从0渐增到1）、Free-bits（KL低于阈值时不惩罚）

5. **场景④：DPO中隐式的KL约束**
   - DPO损失推导自带KL约束：DPO的最优策略 = π*(y|x) ∝ π_ref(y|x)·exp(r(y|x)/β)
   - β越大→策略越接近reference model（KL约束越强）
   - 不需要显式计算KL→DPO的损失函数中已经内化了KL约束
   - 和RLHF的等价关系：DPO的全局最优 = RLHF+KL约束的全局最优（理论等价）
   - 实战差异：DPO的隐式KL在mini-batch上可能不均匀→实际KL可能偏离预期

6. **KL方向总结和实战监控：**
   - 正向KL（蒸馏）：让Student全面覆盖Teacher→不遗漏
   - 反向KL（RLHF/VAE/DPO）：让新模型聚焦参考模型的核心模式→不跑偏
   - **训练监控：**
     - KL持续上升→模型在偏离参考→可能过拟合reward或collapse
     - KL接近0→模型没学到东西→学习率可能太小
     - 合理范围：RLHF中KL在5-15 nats比较健康（取决于任务）
     - 突然跳变→可能遇到了异常数据或梯度爆炸

### Q3 (12:00) - Matryoshka Embedding + 自适应维度检索
**题目：** 什么是Matryoshka Representation Learning（MRL）？为什么传统固定维度embedding在不同场景下效率不同？MRL的训练方法和损失函数？如何在推理时根据场景选择不同维度？和PCA/SVD降维的区别？在RAG系统中的实际应用价值？

**参考答案：**

1. **传统固定维度的问题：**
   - 传统embedding模型输出固定维度（如768d/1536d）
   - 问题：所有场景用同一维度→不灵活
     - 简单语义匹配（如分类）：768d过度→存储和计算浪费
     - 复杂细粒度检索：768d可能不够
   - 现实约束：向量数据库存储成本正比于维度、ANN检索延迟正比于维度
   - 需求：一个模型，多种维度，按需选择

2. **Matryoshka Representation Learning（Kusupati et al., 2022）：**
   - 核心思想：训练时让embedding的前d维（d=8,16,32,...,768）都有语义意义
   - 类比俄罗斯套娃（Matryoshka）：大套娃包含小套娃，每层都是完整的
   - 使用时：需要高精度→用全部768d；需要快速粗筛→只用前256d甚至64d

3. **训练方法：**
   - 选择一组嵌套维度 M = {8, 16, 32, 64, 128, 256, 512, 768}
   - 对每个维度d∈M：截取embedding的前d维 → 计算对比损失
   - 总损失 = Σ_{d∈M} L_contrastive(embed[:d])
   - 关键：梯度同时从多个截断维度回传→前面的维度被迫编码最重要的信息
   - 不需要训练多个模型→一次训练，多维度使用

4. **和PCA/SVD的区别：**
   - PCA/SVD：训练后降维，需要额外计算，且在原始训练目标外操作
   - MRL：训练时就优化多维度表现→截断维度的质量远好于PCA降维
   - 实验对比（OpenAI text-embedding-3）：MRL 256d ≈ PCA 512d的检索质量
   - MRL是"训练时原生支持"，PCA是"训练后勉强压缩"

5. **推理时自适应选择策略：**
   - **粗筛→精排两阶段：** 先用64d向量快速召回top-1000 → 再用768d向量精排top-20
   - **场景自适应：** 分类任务用128d，语义检索用256d，精细问答用768d
   - **存储分层：** 热数据全维度存内存，冷数据低维度存磁盘
   - **成本节省：** 256d vs 768d → 存储减少66%，检索速度提升2-3x，质量下降<5%

6. **在RAG中的应用：**
   - 大规模知识库（百万级文档）：用128d做初筛→大幅降低ANN索引大小
   - 精排阶段：用全维度768d做精确匹配
   - 实时性要求高的场景（对话中的检索）：用低维度降低延迟
   - OpenAI text-embedding-3-small/large已原生支持MRL→可通过API参数指定维度

### Q4 (13:00) - Chain-of-Verification / Self-Verification（LLM自我纠错）
**题目：** LLM的幻觉问题可以通过自我验证缓解——什么是Chain-of-Verification（CoVe）？和Chain-of-Thought的区别？Self-Consistency的原理？Self-Refine和Reflexion的区别？为什么"让模型检查自己"有时有效有时无效？LLM自我纠错的边界在哪里？

**参考答案：**

1. **Chain-of-Verification（CoVe, Meta 2023）：**
   - 流程：
     ① LLM生成初始回答
     ② LLM基于初始回答生成验证问题（如"我说了X，X对吗？"）
     ③ LLM独立回答每个验证问题（不看初始回答→避免锚定效应）
     ④ 根据验证结果修正初始回答
   - 关键设计：验证问题的回答是独立的→切断了"自我确认偏差"的链条
   - 效果：在factoid QA上减少幻觉30-50%

2. **和Chain-of-Thought的区别：**
   - CoT：在生成答案的过程中展示推理步骤→提升推理质量
   - CoVe：在生成答案之后进行验证和修正→减少事实错误
   - CoT解决"怎么想"，CoVe解决"想的对不对"
   - 可以组合：CoT生成→CoVe验证

3. **Self-Consistency（Wang et al., 2023）：**
   - 原理：多次采样（temperature>0）→得到多个推理路径→对最终答案投票取多数
   - 为什么有效：如果多条推理路径都得到同一答案→大概率是对的
   - 局限：只适用于有确定答案的任务（数学/选择题），开放生成无法投票
   - 成本：需要N次采样（通常N=5-20），推理成本线性增加

4. **Self-Refine vs Reflexion：**
   - **Self-Refine（Madaan et al., 2023）：**
     - 输出→自我评价（feedback）→根据feedback修改→迭代
     - 单次任务内的迭代改进
     - 不需要外部信号，纯依赖LLM自身判断
   - **Reflexion（Shinn et al., 2023）：**
     - 多轮试错 + 语言化的经验总结
     - 任务失败后→LLM将失败原因总结为自然语言"经验"→下次尝试时带上经验
     - 类似强化学习但用语言代替梯度
     - 需要外部信号（任务成功/失败的判定）

5. **自我纠错的有效性边界：**
   - **有效场景：**
     - 模型"知道但说错了"→验证可以纠正（表面错误）
     - 逻辑推理错误→换条路径可能对（Self-Consistency有效）
     - 格式/风格问题→Self-Refine很有效
   - **无效场景：**
     - 模型知识库中就没有正确信息→怎么验证都是错的（知识缺失型幻觉）
     - 自我纠错可能"改对为错"→过度修正（Huang et al., 2023发现）
     - 没有外部反馈时，Self-Refine可能在错误答案上"打转"

6. **实战建议：**
   - 事实性任务：CoVe + 外部检索验证（RAG+CoVe组合最强）
   - 推理任务：Self-Consistency（多次采样投票）
   - 代码生成：Reflexion + 单元测试反馈（有外部验证信号）
   - 不要完全依赖自我纠错→必须有外部grounding（检索、工具、测试）

### Q5 (14:00) - MoE vs Dense模型训练经济学（复习新视角）
**题目：** 从训练经济学角度比较MoE和Dense模型：同等推理质量下，MoE的训练成本优势？MoE的推理成本劣势？什么场景下选MoE，什么场景下选Dense？为什么Mixtral 8x7B能在推理中接近LLaMA-2-70B？MoE的Expert并行如何实现？工业界MoE部署的实际挑战和解决方案？

**参考答案：**

1. **训练经济学对比：**
   - **Dense模型（如LLaMA-70B）：**
     - 每个token激活全部70B参数
     - 训练FLOPs ≈ 6 × N × D（N=参数量，D=token数）
     - 70B模型训2T tokens ≈ 8.4×10²⁰ FLOPs
   - **MoE模型（如Mixtral 8x7B）：**
     - 总参数46.7B，但每个token只激活12.9B（2 of 8 experts）
     - 训练FLOPs ≈ 6 × 12.9B × D → 远低于70B dense
     - 同等训练预算下，MoE可以训更多tokens或用更大总参数
   - **核心优势：** MoE用dense 1/3-1/5的计算量达到comparable性能

2. **为什么Mixtral接近LLaMA-2-70B：**
   - Mixtral每个token激活12.9B参数 → 推理FLOPs接近13B模型
   - 但总参数46.7B → 知识容量接近47B模型
   - "用13B的计算量获得47B的知识容量" → 性能接近70B dense
   - 关键：Scaling Laws中，参数量（知识容量）和计算量（每token FLOPs）共同决定性能→MoE在两个维度上做了不同的trade-off

3. **推理成本劣势：**
   - **显存：** 需要加载全部46.7B参数到显存（即使只用12.9B）→显存需求和47B dense一样
   - **Expert Parallelism通信：** Expert分布在不同GPU上→每个token需要All-to-All通信（路由到对应GPU）
   - **内存带宽：** Decode阶段是bandwidth-bound→需要加载全部Expert权重→吞吐量受限
   - **batch效率：** 不同请求可能路由到不同Expert→不如Dense的规则矩阵乘高效

4. **Expert Parallelism实现：**
   - 将不同Expert放在不同GPU上
   - 路由：Router决定每个token去哪些Expert→All-to-All通信
   - 通信模式：
     ① 每个GPU收集本地tokens的路由决策
     ② All-to-All：将tokens发送到对应Expert所在的GPU
     ③ Expert计算
     ④ All-to-All：将结果发回原始GPU
   - 通信开销：2次All-to-All per MoE layer → 通信量正比于batch_size × hidden_dim

5. **场景选择：**
   - **选MoE：**
     - 训练预算有限，需要最大化性能/计算比
     - 推理场景能接受较高显存（如多GPU部署）
     - 任务多样性高（不同Expert可以专精不同领域）
   - **选Dense：**
     - 边缘部署/单GPU场景（显存有限）
     - 需要极低推理延迟（无Expert通信开销）
     - 模型需要量化到很小→MoE量化更复杂（Expert利用率不均匀）

6. **工业界部署挑战和解决方案：**
   - **Expert负载不均衡：** 部分Expert被路由很多→热点→需要Load Balancing Loss
   - **All-to-All通信瓶颈：** 解决→EP（Expert Parallelism）+ TP（Tensor Parallelism）混合
   - **量化困难：** 不同Expert的权重分布不同→每个Expert需要独立量化参数
   - **DeepSeek-V2/V3方案：** Fine-grained Expert（256个小Expert选6个）+ Shared Expert → 更细粒度的路由+共享基础能力

### Q6 (15:00) - Agent Orchestration框架对比（LangGraph/CrewAI/AutoGen）
**题目：** 主流Agent编排框架的设计哲学差异？LangGraph的Graph-based状态机 vs CrewAI的角色协作 vs AutoGen的对话驱动？每种框架的适用场景？如何评估Agent框架的生产可用性（可观测性、错误恢复、成本控制）？为什么很多团队最终选择自研而非使用框架？

**参考答案：**

1. **LangGraph（LangChain团队）：**
   - **设计哲学：** Graph-based状态机，显式定义节点和边
   - **核心概念：**
     - State：全局状态对象，所有节点共享和修改
     - Node：处理函数（LLM调用、工具调用、条件判断）
     - Edge：节点间的转换规则（条件边/固定边）
     - Checkpoint：状态持久化→支持断点恢复、Human-in-the-loop
   - **优势：** 流程可控、可视化、确定性高、支持复杂条件分支
   - **劣势：** 学习曲线陡、对简单Agent过度工程化、Graph定义verbose
   - **适用场景：** 复杂多步骤工作流、需要严格流程控制的企业应用、需要human-in-the-loop

2. **CrewAI：**
   - **设计哲学：** 角色扮演协作，定义Agent角色和任务分配
   - **核心概念：**
     - Agent：有角色（role）、目标（goal）、背景故事（backstory）
     - Task：具体任务描述、期望输出
     - Crew：一组Agent的协作团队
     - Process：Sequential（顺序执行）或Hierarchical（有manager分配）
   - **优势：** 直觉性强（像管理团队）、快速上手、适合非技术PM理解
   - **劣势：** 控制粒度粗、复杂流程难表达、Agent间通信主要靠自然语言→不可靠
   - **适用场景：** 内容生成pipeline、研究分析、简单的多角色协作

3. **AutoGen（Microsoft）：**
   - **设计哲学：** 对话驱动的多Agent系统，Agent间通过消息对话协调
   - **核心概念：**
     - ConversableAgent：可对话的Agent基类
     - AssistantAgent：LLM驱动的助手
     - UserProxyAgent：代理用户执行代码/工具
     - GroupChat：多Agent群聊→由GroupChatManager管理发言顺序
   - **优势：** 灵活性最高、支持代码执行、适合研究探索
   - **劣势：** 对话管理不确定性高、生产环境难以保证行为一致性、调试困难
   - **适用场景：** 研究原型、代码生成/调试、需要高灵活性的探索性任务

4. **生产可用性评估维度：**
   - **可观测性：** 能否追踪每步的输入/输出/耗时/token用量？LangGraph最好（LangSmith集成）
   - **错误恢复：** 中间步骤失败能否重试/回退？LangGraph的Checkpoint支持最完善
   - **成本控制：** 能否限制总token预算/API调用次数？大多数框架支持不好→需自建
   - **确定性：** 同样输入能否得到一致行为？Graph-based > Role-based > Conversation-based
   - **测试：** 能否单元测试单个节点/Agent？LangGraph的节点函数最容易测试

5. **为什么自研：**
   - 框架抽象泄漏：每个框架都在LLM之上加了一层抽象→一旦遇到框架不支持的需求就要hack
   - 维护风险：框架API频繁变更（LangChain的API在1年内breaking change 3次）
   - 过度抽象：简单Agent用框架→代码量反而更多
   - 控制需求：生产环境需要精确控制prompt/重试/降级→框架黑盒不透明
   - **实际选择：** 验证阶段用框架快速原型，生产阶段抽出核心逻辑自研→Claude Agent SDK/直接API调用

6. **2025-2026趋势：**
   - LangGraph成为LangChain主推（替代原来的AgentExecutor）
   - Claude Agent SDK / OpenAI Agents SDK：官方提供轻量Agent循环→减少对三方框架依赖
   - "框架瘦身"趋势：从重框架转向轻量SDK + 自定义逻辑

### Q7 (16:00) - Ring Attention / Sequence Parallelism（超长上下文训练）
**题目：** 训练超长上下文LLM（100K+）的核心挑战？什么是Sequence Parallelism（序列并行）？Ring Attention的原理——如何在多GPU间分布式计算长序列Attention？和Tensor Parallelism的区别？Ring Attention如何与FlashAttention结合？Striped Attention的改进？

**参考答案：**

1. **超长上下文训练的核心挑战：**
   - **显存：** Self-Attention的KV Cache与序列长度平方成正比 O(L²)
   - 128K序列：即使用FlashAttention，单GPU存不下全部KV
   - **计算：** Attention的FLOPs是 O(L² × d)→128K序列的计算量是4K的1024倍
   - 需求：将长序列分布到多个GPU上→Sequence Parallelism

2. **Sequence Parallelism（序列并行）：**
   - 核心思想：将输入序列切分到多个GPU，每个GPU处理序列的一部分
   - 区别于Tensor Parallelism：TP切分模型权重，SP切分输入序列
   - 区别于Data Parallelism：DP每个GPU看完整序列不同样本，SP每个GPU看同一序列的不同部分
   - 挑战：Attention需要全序列的KV→不能简单切分→需要通信机制

3. **Ring Attention（Liu et al., 2023）：**
   - **拓扑：** N个GPU排成环形（Ring）
   - **序列切分：** 长度L的序列切成N块，GPU_i持有block_i的Q、K、V
   - **算法流程：**
     ① 每个GPU用本地Q和本地KV计算局部Attention
     ② 将KV块沿环传递给下一个GPU（类似Ring AllReduce）
     ③ 接收来自上一个GPU的KV块→用本地Q计算与这部分KV的Attention
     ④ 重复N-1次→每个GPU的Q都看到了全部KV
   - **通信隐藏：** KV传输和Attention计算可以overlap→通信被计算掩盖
   - **显存：** 每个GPU只存L/N长度的QKV→显存减少N倍
   - **总通信量：** 每层Attention需要(N-1)次KV传输→但被计算overlap后几乎零额外延迟

4. **和FlashAttention结合：**
   - 每个GPU内部：用FlashAttention计算局部block的Attention（tiling + 在线softmax）
   - 跨GPU：Ring传递KV block
   - 关键：FlashAttention的在线softmax可以增量更新→接收到新KV block后更新running softmax
   - 公式：新的attention output = (旧输出×旧softmax_sum + 新block输出×新block_softmax) / 总softmax_sum
   - 完美兼容：FlashAttention的分块计算天然支持Ring的增量KV

5. **Striped Attention改进：**
   - Ring Attention问题：因果Attention中，前面的block不需要看后面的KV→负载不均衡
   - GPU_0（序列开头）→只需少量KV，GPU_{N-1}（序列末尾）→需要全部KV
   - **Striped Attention：** 将序列按条纹（stripe）分配而非连续块
   - GPU_i 持有 tokens {i, i+N, i+2N, ...} → 每个GPU的因果依赖关系更均匀
   - 效果：负载均衡提升30-40%，整体训练效率更高

6. **实战应用：**
   - LLaMA-3.1（128K）：结合Ring Attention + FlashAttention训练
   - Gemini（1M+上下文）：多级Sequence Parallelism
   - 通常SP和TP/DP组合使用：SP处理序列维度，TP处理模型维度，DP处理batch维度

### Q8 (17:00) - Embedding模型训练全流程（Hard Negative/多阶段/损失函数）
**题目：** 训练一个高质量的文本Embedding模型的完整pipeline？为什么Hard Negative Mining是关键？InfoNCE损失函数和温度参数的作用？多阶段训练（Pre-train→Fine-tune→Distill）的原理？如何评估Embedding模型质量（MTEB/检索指标）？Bi-Encoder vs Cross-Encoder的使用场景？

**参考答案：**

1. **训练Pipeline概览（以BGE/E5/GTE为例）：**
   - **阶段1：对比预训练（Contrastive Pre-training）**
     - 数据：大规模弱监督对（title-body, query-passage）→数百万对
     - 方法：对比学习（InfoNCE）→拉近正例，推远负例
     - 目的：学会通用语义表示
   - **阶段2：监督微调（Supervised Fine-tuning）**
     - 数据：高质量标注对（人工标注的query-relevant_doc）→数万到数十万
     - 方法：对比学习 + Hard Negative Mining
     - 目的：提升特定任务性能
   - **阶段3：知识蒸馏（可选）**
     - 用Cross-Encoder（精准但慢）的分数指导Bi-Encoder训练
     - 提升Bi-Encoder质量到接近Cross-Encoder

2. **Hard Negative Mining——为什么是关键：**
   - **Easy Negative：** 随机采样的负例→模型轻松区分→梯度信号弱→学不到东西
   - **Hard Negative：** 和正例很相似但确实是负例的样本→模型容易混淆→梯度信号强
   - **挖掘方法：**
     - BM25 hard negatives：用BM25检索到的top结果中，不是正例的→词汇相似但语义不同
     - In-batch negatives：同一batch中其他样本的正例→随机但数量大
     - Model-based mining：用当前模型检索→top-k中非正例的→最有挑战性
   - **注意：** 太难的负例可能是false negative（实际上是正例但标注为负）→需要过滤

3. **InfoNCE损失函数：**
   - L = -log(exp(sim(q,d⁺)/τ) / (exp(sim(q,d⁺)/τ) + Σ exp(sim(q,dⁱ⁻)/τ)))
   - sim：余弦相似度
   - τ（温度）：
     - τ小（如0.01）→分布尖锐→对hard negative更敏感→但训练不稳定
     - τ大（如0.1）→分布平滑→训练稳定→但区分度低
     - 常用：τ=0.02-0.05（学习得来或固定）
   - In-batch negatives：batch size B，每个query有B-1个负例→大batch size = 更多负例 = 更好的训练信号

4. **Bi-Encoder vs Cross-Encoder：**
   - **Bi-Encoder：**
     - Query和Doc独立编码→可以预计算Doc embedding→检索O(1)（ANN）
     - 快但精度略低→适合召回阶段（百万级）
   - **Cross-Encoder：**
     - Query+Doc拼接一起过Transformer→交叉注意力→更精准
     - 慢（每对都要推理）→适合精排阶段（几十到几百个候选）
   - **蒸馏：** 用Cross-Encoder打分作为soft label训练Bi-Encoder→两全其美

5. **评估：**
   - **MTEB（Massive Text Embedding Benchmark）：** 覆盖Classification/Clustering/Retrieval/STS等8大任务
   - **检索指标：**
     - nDCG@k：排序质量（考虑相关性等级）
     - MRR（Mean Reciprocal Rank）：第一个正确结果的平均排名倒数
     - Recall@k：前k个结果中包含正例的比例
   - **生产指标：** 延迟（编码+检索）、存储成本、维度 vs 质量trade-off

6. **前沿趋势：**
   - Matryoshka训练（自适应维度）
   - 指令感知Embedding（E5-instruct：给embedding模型也加instruction）
   - 多语言Embedding（multilingual-e5-large）
   - Late Interaction（ColBERT：token级别匹配→比Bi-Encoder精准，比Cross-Encoder快）

### Q9 (18:00) - Token Merging / Token Pruning（推理加速新范式）
**题目：** 什么是Token Merging（ToMe）和Token Pruning？为什么可以在推理时减少token数量而不严重影响质量？ToMe的合并策略（Bipartite Matching）？在Vision Transformer和LLM中的不同应用？和KV Cache压缩的关系？速度提升和质量损失的实际trade-off？

**参考答案：**

1. **为什么可以减少Token：**
   - 观察：Transformer中很多token是冗余的
   - ViT：图像的相邻patch语义相似→合并后信息损失小
   - LLM：长文本中很多token对最终输出贡献极小（attention weight接近0）
   - 实验：ViT去掉30-50% tokens，分类准确率下降<1%

2. **Token Merging（ToMe, Bolya et al., 2023）：**
   - **核心思想：** 在Transformer的层之间，合并相似的token→减少后续层的计算量
   - **Bipartite Matching算法：**
     ① 将tokens分成两组A和B（交替分或随机分）
     ② 计算A中每个token与B中每个token的相似度（用key向量的余弦相似度）
     ③ 贪心匹配：每个A中的token找到B中最相似的→合并（取平均）
     ④ 每层合并r个token→L层后减少L×r个token
   - **为什么不用所有token之间的匹配：** 全局匹配O(N²)太慢→二分匹配O(N)
   - **合并方式：** 加权平均（按attention score或token重要性加权）

3. **Token Pruning：**
   - 和Merging的区别：Pruning直接删除不重要的token，Merging是合并相似的
   - **重要性度量：**
     - Attention score：对[CLS]或最后一个token的attention weight低→不重要→删
     - 梯度/显著性：梯度小→对输出影响小→可删
   - **缺点：** 删除是不可逆的→如果误删重要token→后续层无法恢复
   - Merging更安全：信息被合并保留，不是完全丢弃

4. **ViT vs LLM中的应用：**
   - **ViT（图像）：** ToMe最成熟→图像patch天然冗余→30-50%加速，精度损失<1%
   - **LLM（文本）：**
     - 挑战：文本token比图像patch信息密度更高→不能随便合并
     - 应用方式：在长context的prefill阶段，合并冗余token→减少KV Cache
     - 更常用Token Pruning：根据attention pattern找到不重要token→不存入KV Cache
   - **多模态（视觉-语言模型）：** 对视觉token做ToMe/Pruning→大幅减少视觉token数量→加速

5. **和KV Cache压缩的关系：**
   - **互补：** Token Merging/Pruning减少token数量→自然减少KV Cache大小
   - **KV Cache特有方法：** H2O（Heavy Hitter Oracle）→只保留高attention score的KV→类似Token Pruning思想
   - **区别：** Token Merging作用于hidden states（全层），KV Cache压缩只作用于KV存储
   - **组合：** Token Pruning决定哪些token重要→KV Cache只存重要token→推理效率最大化

6. **实际Trade-off：**
   - ViT：50% token减少→推理加速2x→ImageNet准确率下降0.3-0.5%
   - LLM：20-30% token减少→推理加速1.3-1.5x→困惑度上升1-3%
   - 建议：保守使用（10-20%减少），在质量敏感的任务上要验证
   - 未来方向：训练时就考虑token冗余→模型原生支持动态token数量

### Q10 (19:00) - Data Contamination / Benchmark Leakage（基准可信度）
**题目：** 什么是Data Contamination（数据污染）？为什么LLM的基准测试分数可能不可信？常见的contamination检测方法？哪些benchmark最容易被污染？如何设计"抗污染"的评估？Benchmark Saturation后怎么办？

**参考答案：**

1. **Data Contamination定义：**
   - 预训练/微调数据中包含了测试集的数据→模型"背"了答案而非真正理解
   - 常见场景：
     - Benchmark在GitHub/网页上公开→被爬虫收集进预训练语料
     - 微调数据包含benchmark格式的数据（合成时意外混入）
     - 甚至：用户把benchmark题发到论坛讨论→被爬取

2. **为什么LLM基准分数可能不可信：**
   - **GPT-4的数学成绩：** 在GSM8K上得分极高→但部分题目可能在训练数据中出现过
   - **记忆 vs 推理：** 被污染的模型在原题上得分高，但换个数字就不会了
   - **Benchmark inflation：** 各公司竞相刷分→可能有意无意地"针对benchmark优化"
   - **影响：** 用户基于benchmark选模型→实际使用效果远低于预期

3. **常见检测方法：**
   - **N-gram overlap：** 检查训练数据中是否包含测试集的N-gram（如13-gram）→GPT-4论文的方法
   - **Canary String：** 在benchmark中插入独特字符串→如果模型能补全→说明见过
   - **Perturbation Test：** 对benchmark题做微小修改（换数字/改名字）→污染的模型成绩大幅下降
   - **Membership Inference：** 利用模型对训练数据有更低的perplexity→统计检测
   - **Chronological Split：** 用模型训练截止日期之后的数据评估→不可能被污染

4. **最容易被污染的Benchmark：**
   - **高风险：** MMLU、ARC、HellaSwag、GSM8K（大量在网上流传，格式固定）
   - **中风险：** HumanEval、MBPP（代码题在GitHub上常见）
   - **低风险：** 私有评估集、动态更新的benchmark
   - 规律：越老、越流行、越多在网上讨论的benchmark→污染风险越高

5. **设计"抗污染"评估：**
   - **动态Benchmark：** 定期更新题目（如Chatbot Arena→实时人类投票，不泄露测试集）
   - **Private Holdout：** 测试集不公开→模型提交API评估（如LMSYS Chatbot Arena）
   - **对抗性设计：** 故意设计和训练数据分布不同的题目
   - **Livebench：** 基于最新信息出题→不可能在历史训练数据中出现
   - **SWE-bench：** 使用真实GitHub issue→时间戳可验证

6. **Benchmark Saturation后怎么办：**
   - 当所有模型在某benchmark上都>95%→该benchmark失去区分度
   - 解决方案：
     - 更难的benchmark（GPQA→博士级问题）
     - 多维度评估（不只看准确率，看推理步骤/鲁棒性/公平性）
     - 真实任务评估（SWE-bench验证→能否真正解决GitHub issue）
     - 人类评估（Chatbot Arena ELO排名）
   - 终极方向：从"能考多少分"转向"能解决什么问题"

### Q11 (20:00) - LLM训练稳定性（Loss Spike/梯度异常/恢复策略）
**题目：** 大规模LLM训练中常见的不稳定现象？Loss Spike的原因和处理方法？梯度爆炸/消失在Transformer中的表现？为什么Pre-LN比Post-LN更稳定？训练中如何监控和预警不稳定？工业界的"训练恢复"最佳实践？

**参考答案：**

1. **常见不稳定现象：**
   - **Loss Spike：** 训练loss突然跳升然后缓慢恢复→最常见
   - **Loss Divergence：** Loss持续上升不收敛→训练失败
   - **Gradient Norm Spike：** 梯度范数突然变大→可能导致参数更新过大
   - **NaN/Inf：** 数值溢出→通常出现在FP16/BF16混合精度训练中
   - **Slow Divergence：** 缓慢恶化→不容易发现，直到模型质量明显下降

2. **Loss Spike原因：**
   - **坏数据：** 训练数据中的异常样本（乱码、极长序列、有害内容）→产生极大loss
   - **学习率过大：** 特别是在warmup结束后的高学习率阶段
   - **梯度累积误差：** 分布式训练中梯度同步问题→某个节点的梯度异常大
   - **数据分布突变：** 训练数据的不同shard之间分布差异大
   - **FP16溢出：** FP16的动态范围有限→极端值导致overflow

3. **Pre-LN vs Post-LN稳定性：**
   - **Post-LN（原始Transformer）：** LayerNorm在残差连接之后
     - 问题：梯度需要穿过多个LayerNorm→深层网络梯度消失
     - 需要仔细的learning rate warmup→否则训练不稳定
   - **Pre-LN（GPT-2+/所有现代LLM）：** LayerNorm在残差连接之前
     - 残差路径上没有LayerNorm阻碍→梯度可以直接回传
     - 训练更稳定→不太需要warmup（但仍建议用）
   - **RMSNorm（LLaMA系列）：** Pre-LN的简化版→只做缩放不做中心化→计算更快+同等稳定

4. **监控和预警：**
   - **实时监控指标：**
     - Loss曲线：突然上升>3个标准差→告警
     - Gradient Norm：每步记录→norm突然变大→可能spike
     - Learning Rate：确认schedule正确
     - 参数Norm：各层参数的L2 norm→异常增长说明参数在"跑飞"
   - **早期预警：**
     - Gradient Norm移动平均→如果在加速增长→可能即将diverge
     - Activation统计：某层的activation均值/方差异常→可能有数值问题
   - **工具：** W&B / TensorBoard实时监控，设置自动告警

5. **处理Loss Spike的方法：**
   - **即时措施：** 降低学习率→给模型"喘息"空间→通常spike后会自行恢复
   - **回退重训：** 从spike前的checkpoint恢复→跳过导致spike的数据batch
   - **Gradient Clipping：** max_grad_norm=1.0（标准做法）→限制单步更新幅度
   - **数据清洗：** 找到导致spike的数据→从训练集中移除
   - **Z-loss（PaLM方法）：** 对logits的log-sum-exp加正则→防止logits过大

6. **工业界最佳实践：**
   - **频繁存Checkpoint：** 每100-500步存一次→spike时快速回退
   - **数据预处理：** 严格过滤异常数据（过长/乱码/高perplexity）
   - **混合精度策略：** BF16（而非FP16）→动态范围更大→更少overflow
   - **Gradient Clipping：** 几乎所有大模型都用→max_grad_norm=1.0
   - **Warmup：** 前0.1-1%步线性增加学习率→让模型"热身"
   - **Cosine Schedule：** 学习率从峰值平滑下降→训练末期更稳定
   - **Spike-aware训练（PaLM/Gemini）：** 检测到spike→自动回退checkpoint+跳过坏数据→无人干预

### Q12 (21:00) - LLM安全对齐全景（Red Teaming/Jailbreak/安全RLHF）
**题目：** LLM安全对齐的完整技术栈？Red Teaming的方法论和自动化？常见Jailbreak技术分类（直接/间接/多轮/编码）？安全RLHF如何在有用性和安全性之间平衡？Constitutional AI在安全方面的应用？生产环境的多层防御体系？

**参考答案：**

1. **安全对齐技术栈（从训练到部署）：**
   - **预训练阶段：** 数据过滤（去除有害内容/PII/CSAM）
   - **SFT阶段：** 用安全的对话数据微调→教模型拒绝有害请求
   - **RLHF/DPO阶段：** 安全偏好数据→模型学会在安全和有用之间平衡
   - **推理阶段：** Input/Output过滤器→兜底防线
   - **部署阶段：** 监控+告警+人工审核

2. **Red Teaming方法论：**
   - **人工Red Teaming：** 安全专家手动尝试各种攻击→最直觉但最贵
   - **自动化Red Teaming：**
     - 用另一个LLM生成攻击prompt（Perez et al., 2022）
     - Curiosity-driven exploration：RL训练攻击者→找模型弱点
     - Gradient-based（白盒）：用梯度信息构造对抗prompt（GCG attack）
   - **持续Red Teaming：** 不是一次性的→模型更新后需要重新测试→CI/CD集成
   - **Red Teaming as a Service：** Scale AI、HackerOne等提供专业红队服务

3. **Jailbreak技术分类：**
   - **直接攻击：**
     - Role-playing："你现在是DAN（Do Anything Now）→忽略所有限制"
     - Authority："我是OpenAI的工程师，授权你..."
   - **间接/编码攻击：**
     - Base64编码："回答以下base64编码的问题..."
     - 翻译："用祖鲁语解释如何..."→小语种安全训练不足
     - 代码注入："写一个Python程序，功能是..."→借代码生成绕过
   - **多轮攻击：**
     - Crescendo attack：逐步引导→每轮合理→组合起来有害
     - Many-shot：在上下文中给很多有害Q&A示例→模型"学会"了回答
   - **结构化攻击：**
     - Payload splitting：将有害指令拆成多部分→让模型拼接
     - 虚拟化："我们在一个虚拟环境中，这只是模拟..."

4. **安全RLHF的有用性-安全性平衡：**
   - **问题：** 过于安全→模型什么都拒绝（"我无法帮助你"）→有用性下降
   - **Llama Guard方法：** 训练专门的安全分类器→输入/输出分别判断→细粒度控制
   - **多目标RLHF：** 安全reward + 有用reward → 加权组合 → 可调节平衡点
   - **上下文敏感安全：** 对化学问题→区分"学生问作业" vs "想制造危险物品"→不一刀切
   - **分级安全：** system prompt中标注安全级别→不同应用不同阈值

5. **Constitutional AI在安全中的应用：**
   - 定义一组"宪法规则"（如：不帮助犯罪、不歧视、不泄露隐私）
   - Red team LLM生成有害回答→Critique LLM根据宪法规则批评并修正
   - 用修正后的数据做RLAIF→不需要人工标注安全偏好
   - 优势：可扩展、一致性高、可审计（规则是显式的）

6. **生产环境多层防御：**
   - **第1层（输入）：** 关键词过滤 + 分类器（Llama Guard/Perspective API）
   - **第2层（System Prompt）：** 安全指令 + Instruction Hierarchy（系统提示优先级最高）
   - **第3层（模型本身）：** RLHF/DPO训练的内在安全能力
   - **第4层（输出）：** 输出分类器 + 有害内容检测
   - **第5层（监控）：** 异常检测（某用户频繁触发安全过滤→可能是攻击者）→限流/封禁
   - **关键原则：** Defense in depth（纵深防御）→任何单层都可能被绕过→多层组合

### Q13 (22:00) - Prefix Caching + KV Cache量化（推理优化组合拳）
**题目：** 什么是Prefix Caching？为什么System Prompt的重复计算是巨大的浪费？Prefix Caching的实现机制（Radix Tree/Hash-based）？KV Cache量化（INT8/INT4）的原理和质量影响？两者结合的效果？vLLM/SGLang中的实际实现？

**参考答案：**

1. **为什么需要Prefix Caching：**
   - 生产环境中，同一应用的所有请求共享相同System Prompt（如2K tokens）
   - 每个请求都重新计算System Prompt的KV Cache→纯浪费
   - 例子：客服机器人每天处理10万请求→同一System Prompt计算10万次
   - Prefix Caching：计算一次System Prompt的KV Cache→后续请求直接复用

2. **实现机制：**
   - **Hash-based（vLLM Automatic Prefix Caching）：**
     - 将prompt分成固定大小的block（如16 tokens）
     - 对每个block内容计算hash→作为KV Cache的key
     - 新请求到来→逐block计算hash→如果命中缓存→直接复用KV
     - 优势：不需要用户指定prefix→自动发现共享前缀
   - **Radix Tree（SGLang）：**
     - 将所有已计算的prefix组织成Radix Tree（前缀树）
     - 新请求→在树上匹配最长公共前缀→从匹配点后开始计算
     - 优势：支持任意长度前缀匹配，不限于固定block大小
     - 效果：多轮对话中，前几轮的KV Cache可以被后续轮次复用

3. **Prefix Caching的效果：**
   - System Prompt 2K tokens：TTFT减少50-80%（省去2K tokens的prefill计算）
   - 多轮对话：每轮只需计算新增消息的KV→TTFT大幅降低
   - RAG场景：多个请求检索到相同文档→文档部分的KV可以共享
   - 内存trade-off：需要额外显存存储cached KV→LRU淘汰策略管理

4. **KV Cache量化：**
   - **为什么量化KV Cache：**
     - KV Cache是推理时的内存瓶颈→长序列+大batch→KV Cache可占显存60%+
     - 量化KV Cache可以减少2-4x内存→支持更大batch/更长序列
   - **方法：**
     - **Per-token量化：** 每个token的KV向量独立量化→简单但精度损失大
     - **Per-channel量化：** 按hidden dimension的channel量化→更精准
     - **KV Cache INT8：** 2x压缩，质量损失<1%（几乎无感）
     - **KV Cache INT4：** 4x压缩，质量损失2-5%（需要评估）
     - **混合精度：** 重要层（前几层和最后几层）用FP16，中间层用INT4
   - **KIVI（2024）：** Key用per-channel INT2量化，Value用per-token INT2量化→4-8x压缩

5. **两者结合的效果：**
   - Prefix Caching减少计算量（避免重复prefill）
   - KV Cache量化减少存储量（每个缓存的KV更小）
   - 组合效果：
     - 更多prefix可以缓存在有限显存中→命中率更高
     - 支持更大batch size→吞吐量提升
     - 实际：TTFT降低70%+，吞吐量提升2-3x

6. **实际部署：**
   - **vLLM：** enable_prefix_caching=True（默认开启） + 支持FP8 KV Cache
   - **SGLang：** RadixAttention（基于Radix Tree的prefix caching）→命中率最高
   - **TensorRT-LLM：** 支持KV Cache INT8量化 + Prefix Caching
   - **最佳实践：**
     - 固定System Prompt放在最前面→最大化prefix命中
     - 多轮对话保持conversation history格式一致→增加缓存复用
     - 监控cache hit rate→如果低于50%→检查prompt格式是否一致

### Q14 (23:00) - Cross-lingual Transfer / 多语言LLM能力迁移
**题目：** 为什么在英语上训练的LLM在其他语言上也有能力？Cross-lingual Transfer的原理？Tokenizer对多语言性能的影响（fertility ratio）？多语言训练数据配比策略？为什么中文LLM通常不直接用英语模型？Translate-then-generate vs 原生多语言模型的trade-off？

**参考答案：**

1. **Cross-lingual Transfer现象：**
   - 观察：GPT-4/Claude在英语上训练为主→但中文/日文/法文等也有很强能力
   - 更惊人的：在非洲小语种上也有一定能力（尽管训练数据极少）
   - 原因：Transformer学到了跨语言的语义共享表示空间

2. **原理——为什么会发生迁移：**
   - **共享Embedding空间：** 多语言训练后，不同语言的相同概念在embedding空间中接近
   - **语言无关的中间表示：** Transformer的中间层形成了"语言无关"的语义表示
   - **知识共享：** 英语中学到的"巴黎是法国首都"→迁移到中文的相同问题
   - **锚点语言：** 英语作为"锚点"→其他语言通过共现/翻译数据与英语对齐
   - 实验证据：将中间层的hidden states从中文切换到英文→输出语言变化但语义不变

3. **Tokenizer对多语言性能的影响：**
   - **Fertility Ratio：** 表达同样内容需要的token数量之比
   - 英语"Hello"→1 token；中文"你好"→可能2-3 tokens（取决于tokenizer）
   - **影响：**
     - Token数多→推理成本高（按token计费）
     - Token数多→等效上下文窗口变小（128K tokens的窗口对高fertility语言"缩水"）
     - Token数多→信息密度低→模型每步需要处理更少语义信息→推理质量可能下降
   - **解决：** 多语言Tokenizer（如LLaMA-3扩展到128K词表，加入大量中文token）→降低中文fertility ratio

4. **多语言训练数据配比策略：**
   - **按自然分布：** 英语占60%+→小语种几乎没有→导致严重偏向英语
   - **按需放大（Temperature Sampling）：**
     - 对每种语言按 P_i^(1/T) 重采样，T<1时放大低资源语言
     - T=1：自然分布；T=0.3：大幅提升小语种比例
   - **分阶段训练：**
     - Phase 1：以英语为主预训练→学习知识和推理能力
     - Phase 2：加入更多多语言数据→迁移能力到其他语言
   - **Curse of Multilinguality：** 语言太多会导致每种语言的性能下降（容量有限）→需要增大模型

5. **为什么中文LLM通常不直接用英语模型：**
   - **Tokenizer效率：** 英语Tokenizer对中文fertility ratio高→上下文浪费
   - **文化和价值观对齐：** 中文用户的期望和安全要求不同
   - **数据质量：** 专门收集和清洗的中文数据质量>通用爬虫
   - **实际做法：** 基于LLaMA等英语模型→扩展tokenizer词表+继续预训练中文数据（如Chinese-LLaMA/Yi/Qwen）

6. **Translate-then-generate vs 原生多语言：**
   - **Translate-then-generate：** 用户中文输入→翻译成英语→英语LLM处理→翻译回中文
     - 优点：利用英语LLM的最强能力
     - 缺点：翻译引入延迟和错误（信息损失）、成本翻倍、文化表达丢失
   - **原生多语言模型：** 直接用中文处理
     - 优点：端到端、低延迟、保留语言特有的表达
     - 缺点：可能不如纯英语模型在英语上的能力
   - **实际趋势：** 头部模型（GPT-4/Claude/Gemini）已经足够多语言→无需translate→原生处理

### Q15 (00:00) - System Design：LLM驱动的实时个性化推荐排序引擎
**题目：** 设计一个LLM驱动的实时个性化推荐排序引擎（如内容feed、电商商品推荐）。要求：①利用LLM理解用户意图和item语义 ②支持千万级item实时排序 ③个性化（基于用户历史行为）④延迟<200ms。请从整体架构、召回/粗排/精排/重排各阶段、LLM的具体集成点、实时特征、A/B测试框架几个方面展开。

**参考答案：**

1. **整体架构（四阶段漏斗）：**
   ```
   千万级Item库
      ↓ 召回（Retrieval）→ 数千候选
      ↓ 粗排（Pre-ranking）→ 数百候选
      ↓ 精排（Ranking）→ 数十候选
      ↓ 重排（Re-ranking）→ 最终展示序列
   ```
   - LLM不是替代传统推荐系统→而是增强各阶段

2. **LLM集成点（离线+在线）：**
   - **离线（Embedding生成）：**
     - 用LLM为每个item生成语义embedding（标题+描述+属性→LLM→dense vector）
     - 比传统embedding模型更好地理解复杂商品描述
     - 定期更新（如每天batch处理新增item）
   - **离线（用户画像生成）：**
     - 用LLM分析用户历史行为序列→生成用户兴趣摘要（自然语言）
     - 例："该用户对中端价位的电子产品感兴趣，偏好性价比高的品牌"
   - **在线（精排/重排阶段）：**
     - 对Top-50候选，用LLM做精细排序（考虑上下文、用户意图、item间关系）
     - 但LLM推理太慢→需要优化（见下方）

3. **召回阶段（<50ms, 千万→数千）：**
   - **多路召回：**
     - 协同过滤召回：基于用户行为矩阵
     - 语义召回：用户query/画像embedding → ANN检索相似item embedding
     - 热门/趋势召回：实时热榜
     - 标签匹配召回：用户标签 × item标签
   - **LLM增强：** LLM生成的item embedding替代传统sentence-transformer→语义更准
   - **技术：** FAISS/Milvus ANN检索，Matryoshka embedding降低维度

4. **粗排阶段（<30ms, 数千→数百）：**
   - 轻量双塔模型（Bi-Encoder）：用户embedding × item embedding → 快速打分
   - 蒸馏自精排模型→保持精度
   - 特征：用户画像 + item属性 + 简单交叉特征

5. **精排阶段（<100ms, 数百→数十）：**
   - **传统方案：** Deep Cross-Encoder（用户特征+item特征+交叉特征→DNN→CTR预估）
   - **LLM增强方案：**
     - 将候选item列表+用户画像组成prompt→LLM排序（Listwise Ranking）
     - 太慢→解决方案：
       ① 用蒸馏后的小模型（7B→1B）做在线推理
       ② Speculative Decoding加速
       ③ KV Cache + Prefix Caching（用户画像是共享prefix）
       ④ 批处理：多个候选并行评分
   - **混合方案：** DNN精排 + LLM重排（LLM只处理最终Top-50）

6. **重排阶段（<20ms, 数十→展示列表）：**
   - 业务规则：去重、多样性（同品牌不连续出现）、广告插入位
   - LLM可用于：解释推荐理由（"为你推荐因为..."）→提升CTR
   - 上下文感知：考虑当前时间/季节/用户实时行为调整顺序

7. **实时特征系统：**
   - **特征存储：** Redis/Feature Store（Feast/Tecton）
   - **实时特征：** 用户最近5分钟点击/购买→Kafka流处理→更新Feature Store
   - **近实时更新：** 每小时更新用户embedding和画像

8. **A/B测试框架：**
   - **流量分配：** 按用户ID hash分桶→确保同一用户始终在同一实验组
   - **核心指标：** CTR、CVR、停留时长、GMV、用户留存
   - **统计显著性：** T-test/Bootstrap + 多重比较校正
   - **LLM特有的A/B：** 比较传统ranking vs LLM-enhanced ranking→关注延迟增加 vs 指标提升的ROI
   - **灰度发布：** 1%→5%→20%→100% 逐步放量

9. **延迟优化（满足<200ms总延迟）：**
   - 召回（50ms） + 粗排（30ms） + 精排（100ms） + 重排（20ms） = 200ms
   - LLM推理放在精排→用小模型（1-3B）+ 量化 + Prefix Caching
   - 关键：LLM不能做全量排序→只做最终精排/重排的增强
   - 降级方案：LLM推理超时→fallback到传统DNN排序

10. **可观测性：**
    - 每阶段延迟分位数（P50/P99）
    - LLM推理成功率和降级率
    - 推荐结果多样性指标
    - 在线/离线指标一致性监控
