### Q1 (10:00) - PPO四模型原理
**题目：** PPO的原理？从四个模型（Actor/Critic/RM/Reference）讲，描述训练流程和损失函数参数含义。
**shen的回答：**
Actor是要被更新的策略模型；Critic计算advantage，训练中更新；RM提前训练好打分（冻结）；Reference Model给Actor做参考（冻结）。
训练流程：先训练RM（人工标注）；推理时Actor输出response，Critic对每个token预测value，所有value加上RM的分数得到advantage；目标函数两部分：PPO-Clip（限制新旧策略比例r_t不超出0~0.2范围）+ KL惩罚项（控制新旧模型预测差异力度）。

**点评：** 7/10
**✅ 答对的部分：** 四个模型的角色和冻结关系理解正确；训练大流程清晰；PPO-Clip和KL惩罚的目的都说到了。
**❌ 遗漏/不准确的地方：**
1. *Advantage计算公式说错了。* 不是"所有value加起来加上RM分"。正确是：A_t = r_T（RM给最终token打的分）+ γV(s_{t+1}) - V(s_t)，即TD误差，通常用GAE（广义优势估计）平滑。Critic预测的是V(s)，advantage是用RM reward和V的差来算的，不是简单相加。
2. *clip范围说错了。* ε=0.2，clip范围是 (1-ε, 1+ε) = **(0.8, 1.2)**，是对比率r_t(θ)做clip，不是"0到0.2"。
3. *r_t(θ) 没有定义。* r_t(θ) = π_θ(a_t|s_t) / π_θ_old(a_t|s_t)，即新旧策略在同一动作上的概率比值，这是PPO的核心。
4. *Reference Model的具体用途没说完整。* Reference Model是用来计算KL散度的：KL(π_θ || π_ref)，直接约束Actor不要偏离SFT模型太远，防止reward hacking。
5. *没有提到GAE。* 实践中advantage用GAE计算（λ参数控制偏差-方差权衡），面试中提到加分。

**标准损失函数：**
L_PPO = E[min(r_t·A_t, clip(r_t, 1-ε, 1+ε)·A_t)] - β·KL(π_θ || π_ref)

### Q2 (11:00) - Attention数学本质+Scaling
**题目：** Attention(Q,K,V)完整公式、Q/K/V信息检索角度理解、为什么除以√d_k、Multi-Head好处。
**shen的回答：**
公式：Q·K内积除以√d，再乘以V。Q像问题，K像图书馆（看哪本书合适），V是书的内容。√d_k是因为dimension大时QK值域分散，softmax会接近one-hot，绝大部分变0导致梯度消失。Multi-head把dimension拆分，每个head理解query的一部分，提供更多角度理解query。

**点评：** 7/10
**✅ 好的地方：** 公式大体正确；Q/K/V的图书馆类比非常直观；√d_k的梯度消失问题说到了；Multi-head多角度理解的思路对。
**❌ 遗漏/不准确：**
1. *公式写不完整。* 缺少softmax和转置：Attention(Q,K,V) = softmax(QK^T / √d_k) · V，K要转置才能做矩阵乘法。
2. *√d_k的数学原理没说透。* 更精确的解释：假设Q/K每个元素是均值0方差1的随机变量，则Q·K^T的每个元素是d_k个乘积之和，方差为d_k，标准差为√d_k。不除的话点积值域会随d_k增大而增大，导致softmax进入饱和区（梯度趋近0）。
3. *Multi-head机制没说到投影。* Multi-head不只是"拆维度"，每个head有独立的W_Q/W_K/W_V投影矩阵，学习不同的子空间表示（语法/语义/指代等），最后concat后经W_O投影回原维度。
4. *K的类比可以更准确。* K更像图书馆目录（索引），V才是书的内容；K和V是一一对应的。

**标准公式：**
Attention(Q,K,V) = softmax(QK^T / √d_k) · V
MultiHead = Concat(head_1,...,head_h) · W_O，每个head_i = Attention(QW_i^Q, KW_i^K, VW_i^V)

### Q3 (12:00) - KL散度vs交叉熵
**题目：** KL(P||Q)公式和直觉含义、与交叉熵H(P,Q)的数学关系、PPO和DPO中KL的用法、能否用交叉熵替换KL。
**shen的回答：**
KL散度是衡量两个分布之间的差异性。第二小问不会。PPO和DPO中利用KL衡量新旧模型对当前动作预测差异，做惩罚防止参数改动过大。第四问不会。

**点评：** 4/10
**✅ 好的地方：** KL散度基本定义正确；PPO中KL作为惩罚项防止Actor偏离Reference的核心思路对。
**❌ 遗漏/需要补充：**
1. *没有给出公式。* KL(P||Q) = Σ P(x)·log(P(x)/Q(x)) = E_P[log P/Q]，非负，P=Q时为0。
2. *第二问是高频考点。* H(P,Q) = H(P) + KL(P||Q)，即交叉熵 = 信息熵 + KL散度。当P固定时，最小化交叉熵等价于最小化KL。KL不对称是因为KL(P||Q) = Σ P·log(P/Q)而KL(Q||P) = Σ Q·log(Q/P)，两者加权的分布不同。
3. *PPO和DPO中的KL用法不同，不能混为一谈。* PPO：在损失函数中显式添加KL(π_θ || π_ref)作为惩罚项（或约束）；DPO：KL是隐式的，推导过程用了KL约束，但最终损失只需要计算log(π_θ/π_ref)的比值，不显式算KL。
4. *第四问：PPO的KL不能简单换成交叉熵。* 原因：交叉熵需要固定的目标分布（one-hot标签），但PPO中的reference model输出是连续分布，不是one-hot。特殊情况：SFT中标签是one-hot时，CE = -log π_θ(y) ≈ KL(δ_y || π_θ) + const，此时两者等价；但RL训练中不等价。

### Q4 (13:00) - 混合检索BM25+向量
**题目：** BM25算法原理、向量检索原理、两者如何融合（Hybrid Search）、RRF算法。
**shen的回答：**
混合检索在RAG过程中通过不同方式检索相关文档：关键词检索（BM25）、语义相似度检索（向量）、Graph RAG（有层级架构的信息）。检索后可通过rule或reranker精简文档再放入LLM。询问RRF是什么。

**点评：** 5/10
**✅ 好的地方：** 核心思路对，列举了BM25/向量/GraphRAG三种方式，提到reranker精选，知识面广。
**❌ 遗漏：**
1. *没说两者各自的优缺点（这是核心）。* BM25：精确关键词匹配强（型号/代码/专有名词），但不理解语义/同义词；向量检索：语义理解强（同义/改写），但对精确关键词匹配弱。混合检索正是解决各自的短板。
2. *RRF（Reciprocal Rank Fusion）不会。* 公式：RRF(doc) = Σ 1/(k + rank_i)，k通常取60。把BM25和向量检索各自的排名列表融合，不需要归一化分数，鲁棒性强。
3. *没有举具体例子。* BM25适合："iPhone 15 Pro Max价格"（精确型号）；向量检索适合："推荐一款拍照好的手机"（语义理解）。
**RRF示例：**
doc A: BM25 rank=1, 向量 rank=3 → 1/(60+1) + 1/(60+3) = 0.0163+0.0159 = 0.0322
doc B: BM25 rank=2, 向量 rank=1 → 1/(60+2) + 1/(60+1) = 0.0161+0.0163 = 0.0324 → B排第一

### Q7 (16:00) - vLLM推理加速+PagedAttention
**题目：** vLLM的PagedAttention原理、Continuous Batching、KV Cache管理、与普通推理框架的区别。
**shen的回答：** 未作答

**参考答案：**
1. *PagedAttention原理：* 传统推理框架为每个请求预分配一块连续的显存来存储KV Cache，但实际生成长度未知，导致大量显存碎片和浪费（实测浪费率60-80%）。PagedAttention借鉴操作系统虚拟内存的分页机制：将KV Cache分成固定大小的block（如16个token一块），用block table（页表）维护逻辑位置→物理显存的映射，按需动态分配物理block。核心好处：
   - 消除显存碎片：block可以不连续存储，不需要预留最大长度的连续空间
   - 支持Copy-on-Write：多个beam search路径或parallel sampling可共享前缀的KV Cache block，只在分叉时复制被修改的block，显存节省与共享前缀长度成正比
   - 显存利用率接近100%：实测从传统框架的20-40%提升到>96%，同等显存下可服务2-4x更多并发请求
   - 技术细节：Attention计算时通过block table查找物理地址，用自定义CUDA kernel在不连续的block上高效计算Attention（kernel内部处理地址跳转，对计算逻辑透明）
2. *Continuous Batching（连续批处理）：* 传统Static Batching的问题：一批请求必须等最长的那条生成完毕后才能释放GPU资源，短请求生成完后空转等待（padding浪费），GPU利用率极低。Continuous Batching以iteration（单个decode step）为调度粒度：
   - 每个decode step结束后，调度器检查：已完成的请求立即移除并返回结果、等待队列中的新请求立即插入batch
   - 不再有"batch级别的同步等待"，GPU几乎不空转
   - 效果：吞吐量提升2-5x（论文实测），尾延迟也大幅降低
   - 实现关键：需要PagedAttention配合——新请求插入时只需分配新block，不需要重新组织整个batch的KV Cache内存布局
3. *KV Cache管理全生命周期：*
   - Prefill阶段：收到请求后，一次性并行计算整个prompt所有token的KV，存入按需分配的block中。这个阶段是compute-bound（计算密集），耗时与prompt长度成正比
   - Decode阶段：每生成一个新token，计算其K/V并追加到当前block末尾；当前block满了就从空闲池分配新block并更新block table。这个阶段是memory-bound（显存带宽密集），因为每步只算1个token但要读取所有历史KV
   - 回收：请求完成（遇到EOS或达到max_tokens）后，所有block归还空闲池，立即可被新请求复用
   - Prefix Caching（vLLM 0.3+）：相同system prompt的多个请求可共享前缀KV Cache block（通过hash匹配前缀内容），避免重复Prefill计算。在system prompt很长（如RAG场景塞入大量文档）时节省大量计算
   - KV Cache量化：支持FP8/INT8量化KV Cache，显存占用减少50%，精度损失极小（<0.5% on benchmarks）
   - 显存占用估算：以7B模型、32层、128 head_dim、FP16为例，单个token的KV Cache = 2(K+V) × 32层 × 128维 × 2字节 ≈ 16KB；2048长度序列 ≈ 32MB；批量处理100个请求 ≈ 3.2GB
4. *与普通推理框架的对比：*
   - HuggingFace Transformers：单请求顺序推理，无batching优化，KV Cache连续分配且预留最大长度，适合研究原型但不适合生产。吞吐量是vLLM的1/14到1/24
   - TGI（Text Generation Inference）：HuggingFace的生产方案，有Continuous Batching但无PagedAttention，显存效率不如vLLM
   - TensorRT-LLM（NVIDIA）：编译优化+kernel融合+量化，单请求延迟可能最低，但部署复杂（需要针对具体GPU型号编译engine），灵活性差
   - vLLM：PagedAttention + Continuous Batching + 开箱即用（pip install vllm即可），在高并发场景下吞吐量最优。与HF相比吞吐量提升14-24x
   - 关键性能指标：TTFT（Time-to-First-Token，首token延迟，衡量Prefill速度）、TPOT（Time-per-Output-Token，每token延迟，衡量Decode速度）、Throughput（tokens/s，衡量总吞吐）

**面试常见追问：**
- "PagedAttention有什么缺点？"——自定义kernel开发成本高，不连续内存访问可能有少量cache miss开销（但远小于节省的显存收益）；block size选择需要trade-off（太大浪费内部碎片、太小页表开销大）
- "Prefill和Decode为什么要分开优化？"——Prefill是compute-bound（大量并行矩阵运算，GPU计算单元是瓶颈），Decode是memory-bound（每步只算1个token但要读大量KV Cache，显存带宽是瓶颈），优化策略完全不同。新趋势：Disaggregated Serving把Prefill和Decode放在不同GPU上分别优化
- "vLLM支持多GPU吗？"——支持tensor parallelism（模型并行，切分权重矩阵到多卡）和pipeline parallelism（流水线并行，切分层到多卡），可跨多GPU/多节点部署大模型
- "Speculative Decoding是什么？"——用小模型（draft model）快速生成K个候选token，大模型一次性验证，接受率高时等效加速K倍。vLLM已原生支持

### Q8 (17:00) - MoE架构+Router
**题目：** MoE相比FFN的结构变化、Router Top-K流程、负载均衡（Routing Collapse）、Mixtral 8×7B参数量。
**shen的回答：**
有多个专家网络，每个是DNN；通过softmax控制，Top-K代表probability Top-K的专家被激活；负载均衡不会；实际计算量取决于激活的专家数量。

**点评：** 4/10
**✅ 好的地方：** 多专家网络+softmax选Top-K的基本思路对；实际计算量只算激活专家这点答对了。
**❌ 遗漏：**
1. *结构变化没说清楚。* MoE是把Transformer里的FFN层替换成：Router（线性层+softmax）+ N个Expert FFN。每个Expert是独立的FFN（不是泛指DNN），Attention层保持不变。
2. *Router输出是加权求和，不只是"激活"。* 完整流程：g = softmax(x·W_r)，选Top-K得到权重，output = Σ g_i · Expert_i(x)，权重直接参与计算，不是简单开关。
3. *负载均衡（Routing Collapse）是高频考点。* 问题：Router倾向于总路由到相同几个Expert，其余Expert得不到训练→参数浪费。解法：辅助损失 L_aux = α·Σ f_i·P_i（f_i=实际路由比例，P_i=平均路由概率），鼓励均匀分配；另外设置Expert Capacity（每个Expert最多处理多少token）。
4. *Mixtral 8×7B参数量要说清楚。* 总参数≈46.7B（不是8×7=56B，因为Attention层只有一套是共享的）；每个token激活Top-2 Expert，实际计算量≈12.9B，相当于用12B的计算量享受46B的知识容量。

### Q9 (18:00) - Agent死循环解决方案
**题目：** Agent执行中死循环的原因、检测方法、防止机制、工程实现。
**shen的回答：** 未作答（直接要了答案）

**参考答案：**
1. *死循环的常见原因：*
   - 工具调用失败但模型不放弃：工具返回错误信息，模型不理解错误含义或不知道替代方案，反复用相同参数重试。例如API返回403但模型不断重新调用、文件路径不存在但模型反复尝试读取
   - 目标不可达但无退出条件：用户指令模糊（如"优化这段代码"没有明确完成标准），模型永远觉得可以继续改进；或任务本身不可能完成（如要求访问不存在的数据库表）
   - 自我矛盾/振荡：模型在一步中做出修改A，下一步反思后觉得A有问题撤回，然后再做A，形成"修改→撤回→修改→撤回"的振荡。常见于代码修复场景
   - 状态感知失败：模型误解工具返回结果（如把错误信息当作成功），基于错误认知继续执行，永远无法达到预期状态
   - 上下文窗口溢出导致遗忘：长时间运行后，早期的操作历史被截断，模型"忘记"自己已经尝试过某种方法，重新尝试相同路径
2. *检测方法：*
   - Action重复检测：维护最近N步的action fingerprint（tool_name + 参数hash），如果滑动窗口内相同fingerprint出现超过K次（如3次），判定为循环。相似度检测可用：完全匹配、编辑距离低于阈值、或参数的Jaccard相似度高于0.9
   - Output/Observation重复检测：对比最近N步的工具返回结果，如果连续3次返回相同的错误信息或相同的状态描述，说明执行没有产生任何进展
   - 状态不变检测：维护环境状态的摘要或hash（如文件内容hash、测试通过数、数据库记录数），如果执行了action但状态hash不变，标记为无效步骤；连续M个无效步骤触发循环警告
   - 进度指标监控：定义可量化的进度指标（如测试通过数从3→5算进展、编译错误数从10→8算进展），如果连续N步指标无改善则判定停滞
   - LLM自检（辅助手段）：每隔K步让LLM自我评估"是否在重复之前的操作"，但不能作为唯一依据（LLM可能也陷在同样的思维模式中，给出虚假的"没有重复"判断）
3. *防止机制（架构层面）：*
   - 最大步数限制（Max Steps）：硬性上限，如最多执行30个tool call。到达上限后强制停止并返回当前最佳结果。这是最基本也是最必要的兜底机制，所有Agent框架都应有此设置
   - Token/成本预算：限制总消耗token数（包括输入+输出）或API调用成本，防止无限消耗资源。与Max Steps互补——有些步骤token消耗大但步数少
   - 渐进式退出策略：第一次检测到循环→注入反思提示（"你似乎在重复操作，请分析原因并换一种方法"）；第二次→强制约束（"禁止再使用X工具/X参数，请用替代方案"）；第三次→强制停止并汇报当前状态和遇到的障碍
   - 反思机制（Reflection）：每隔K步（如每5步）强制插入一个反思步骤："回顾你过去5步的操作和结果，评估是否有实质进展，如果没有请制定完全不同的策略"。ReAct框架的Thought步骤天然包含这个功能，但需要显式强化"检测循环"的要求
   - 子目标分解与隔离：把复杂任务拆成独立子目标（sub-goals），每个子目标有独立的步数限制和完成条件。子目标A失败不阻塞子目标B的执行，避免单点死循环导致整体卡死
   - 黑名单/冷却机制：连续失败的action（tool+参数组合）加入本次会话的临时黑名单，强制LLM探索新路径。或设置冷却期（同一action至少间隔N步才能再次使用）
   - 多策略回退（Fallback）：主策略失败N次后，自动切换到备选策略（如从"自动修复"切换到"生成修复建议让人工执行"）
4. *工程实现：*
   - 主循环中的检测逻辑：
     ```
     action_history = []  # 存储 (tool_name, param_hash, result_hash)
     for step in range(MAX_STEPS):
         action = agent.plan()
         fingerprint = hash(action.tool + sorted(action.params))
         # 重复检测
         recent = action_history[-WINDOW_SIZE:]
         repeat_count = sum(1 for a in recent if a.fingerprint == fingerprint)
         if repeat_count >= REPEAT_THRESHOLD:
             if escalation_level == 0:
                 inject_reflection_prompt()
             elif escalation_level == 1:
                 blacklist_action(fingerprint)
             else:
                 return force_stop_with_summary()
             escalation_level += 1
             continue
         result = execute(action)
         action_history.append((fingerprint, hash(result)))
     ```
   - 分层超时机制：单个tool call超时（如30s，防止工具本身挂起）→ 单个子任务超时（如5min）→ 整个任务超时（如30min），每层独立处理并有不同的降级策略
   - 优雅降级与汇报：死循环检测触发后，不是简单报错，而是：①保存当前已完成的部分成果②生成结构化的进度摘要（已完成X、卡在Y、尝试过Z方法）③向用户说明具体卡在哪里和建议的手动操作步骤
   - 日志与可观测性：记录每步的action/observation/思考过程/检测指标，便于事后分析死循环根因。关键监控指标：平均步数/任务、循环检测触发率、强制停止率、人工介入率
   - 上下文窗口管理：长时间运行的Agent上下文会膨胀导致截断。解法：定期摘要压缩历史（保留关键决策点和结果，去掉中间的工具原始输出），在摘要中显式包含"已尝试过的方法列表"防止遗忘导致重复

**面试常见追问：**
- "如何区分'合理重试'和'死循环'？"——合理重试通常伴随参数变化（如换不同的搜索关键词、修改不同的代码行）或等待外部状态变化（如等待API限流恢复后重试）；死循环是相同action+相同（或等价）结果的重复。工程上通过参数相似度阈值区分，阈值需要根据场景调优
- "反思机制对弱模型有效吗？"——对强模型（GPT-4/Claude）有效，反思后确实能换策略；对弱模型可能无效（反思后还是选择相同操作）。因此反思只能作为辅助手段，硬性步数限制和黑名单机制是必须的兜底
- "实际项目中最常见的死循环场景？"——代码修复Agent：测试失败→修改代码→还是失败→改回去→再改→循环。根因通常是LLM不理解真正的bug原因，在症状层面反复尝试。解法：在反思步骤中强制要求"先分析错误根因再修改"，或提供更详细的错误上下文（如堆栈跟踪、相关代码片段）
- "与人工客服的escalation有什么区别？"——Agent的escalation应该是结构化的：不只是"我不会"，而是"我尝试了方法A/B/C，卡在X环节，具体错误是Y，建议人工做Z"。这样人工接手时效率最高

### Q10 (19:00) - 长序列Attention O(L²)方案
**题目：** 标准Attention O(L²)瓶颈、FlashAttention原理、稀疏Attention、GQA/MQA、Mamba/SSM。
**shen的回答：**
1. Q/K要算pairwise内积所以O(L²)；2-4不会；5. 稀疏Attention只和附近或随机element算QK；其余不会。

**点评：** 3/10
**✅ 好的地方：** pairwise内积导致O(L²)正确；稀疏Attention的局部+随机思路对。
**❌ 遗漏（4个不会的答案）：**
1. *O(L²)真正瓶颈是内存，不只是计算。* Attention矩阵是L×L，存储需要O(L²)显存。L=128K时矩阵有160亿个元素，远超显存上限——这才是长序列的核心瓶颈。
2. *FlashAttention：IO优化，不降时间复杂度。* 核心思想：分块（Tiling）计算，不把完整L×L矩阵写入HBM（慢）显存，而是在SRAM（快）中分块计算并累积结果。时间复杂度仍是O(L²)，但显存降到O(L)，实际速度提升2-4x。面试常考："FlashAttention降低了复杂度吗？——没有，降低的是显存占用和IO次数。"
3. *GQA/MQA：减少KV Cache大小，加快解码。* MQA：所有Attention头共享一套K/V，Q仍per-head；GQA：分组共享（Llama 3/Mistral用GQA）。作用不是降低训练复杂度，而是推理时KV Cache体积缩小，可支持更大batch或更长序列。
4. *Mamba/SSM：真正O(L)的替代方案。* 用状态空间模型替代Attention，推理时复杂度O(L)（循环更新固定大小状态）；训练时用并行扫描O(L log L)。代价：所有历史压缩进固定状态，不能像Attention那样精确随机访问历史token。

### Q15 (00:00) - LLM as a Judge ❌ 未发送（API限流）
**题目：** LLM as a Judge 是什么？解决了什么问题？相比人工评估有哪些优劣？主要有哪三种评估形式（Pointwise/Pairwise/Reference-based）？实际使用中有哪几种偏差（Bias）？如何缓解？如何验证 Judge 的可靠性？用什么指标衡量与人工的一致性？
**shen的回答：** 未发送（API限流，midnight触发时已达速率上限）

**参考答案：**
1. *定义与解决的问题：* 用LLM替代人工评估LLM输出质量。解决人工评估成本高、速度慢、难以规模化的问题。优势：快速、低成本、可复现；劣势：存在系统性偏差、对微妙质量差异判断不如人类、可能被对抗攻击欺骗。
2. *三种评估形式：*
   - Pointwise：对单个输出直接打分（1-5分），适合绝对质量评估
   - Pairwise：比较两个输出哪个更好（A>B/B>A/平手），更稳定可靠，消除绝对分数的校准问题
   - Reference-based：对比模型输出与参考答案的一致性，适合有标准答案的场景（事实性QA）
3. *常见偏差（Bias）：*
   - Position bias：倾向选择第一个（或最后一个）出现的答案
   - Verbosity bias：倾向选择更长更详细的回答，即使内容质量不高
   - Self-enhancement bias：倾向给自己（同系列模型）生成的内容打高分
   - Style bias：偏好特定格式（如列表、markdown）而非内容质量
   - 缓解方法：交换位置取平均（消除position bias）、多Judge投票、few-shot校准、CoT评估
4. *可靠性验证指标：*
   - Cohen's Kappa / Fleiss' Kappa：与人工标注的一致性（>0.6为substantial agreement）
   - Spearman/Kendall rank correlation：排序一致性
   - Agreement rate：简单一致率
   - 人工评估子集对照：抽样验证Judge与人工评分的分布差异
