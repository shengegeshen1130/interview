### Q1 (10:00) - 向量数据库内部原理（HNSW/IVF/PQ/ScaNN）
**题目：** 向量数据库是RAG系统的核心组件——ANN（近似最近邻）检索的主流算法有哪些？HNSW的多层图结构如何工作？IVF（倒排文件索引）的原理？PQ（Product Quantization）如何压缩向量？ScaNN的改进？不同算法的Recall-QPS-Memory三角trade-off？如何为RAG项目选择向量数据库？

**参考答案：**

1. **ANN检索的必要性：**
   - 精确最近邻：暴力搜索O(N·d)→百万级向量×768维→每次查询需数秒→不可接受
   - ANN：牺牲少量精度（Recall 95-99%）→速度提升100-1000x→毫秒级检索
   - 核心trade-off：Recall（准确率）vs QPS（查询速度）vs Memory（内存占用）

2. **HNSW（Hierarchical Navigable Small World, Malkov 2018）：**
   - **结构：** 多层图→顶层稀疏（远距离跳跃）→底层密集（精确搜索）
   - **构建：**
     - 每个向量随机分配一个最大层级（指数分布→大多数在底层）
     - 从最高层开始，每层找到最近邻并建立双向边
     - 每层保持M个连接（M通常=16-64）
   - **搜索：**
     ① 从顶层的入口点开始
     ② 在当前层贪心地移向query最近邻
     ③ 无法更近时→下降一层→继续搜索
     ④ 到达底层→返回top-k结果
   - **优势：** Recall极高（>99%可达），查询速度快
   - **劣势：** 内存占用大（需要存图结构+原始向量），构建速度慢
   - **适用：** 内存充足、对Recall要求高的场景

3. **IVF（Inverted File Index）：**
   - **原理：** 将向量空间用K-Means聚类为N个簇→每个簇一个倒排列表
   - **搜索：**
     ① 找到query最近的nprobe个簇中心
     ② 只在这nprobe个簇内暴力搜索→大幅缩小搜索范围
   - **参数：** nlist（簇数，通常√N到4√N）、nprobe（搜索簇数，通常1-10%的nlist）
   - **优势：** 内存效率高（只存向量+簇分配）、适合大规模
   - **劣势：** Recall取决于nprobe→nprobe大→慢但准
   - **常见组合：** IVF + PQ（IVF缩小范围 + PQ压缩向量→内存极低）

4. **PQ（Product Quantization）：**
   - **原理：** 将d维向量切分为m个子空间（如768d→8×96d）→每个子空间用K-Means量化为256个中心→每个子向量用1 byte编码
   - **压缩率：** 768维FP32（3072 bytes）→ 8 bytes → 压缩384倍
   - **距离计算：** 预计算query和每个子空间中心的距离表→查表求和→极快
   - **ADC（Asymmetric Distance Computation）：** query不量化，只量化数据库向量→更精确
   - **OPQ（Optimized PQ）：** 先旋转向量空间使子空间更独立→提升量化质量

5. **ScaNN（Google, 2020）：**
   - **核心改进：** 各向异性量化（Anisotropic Quantization）
   - 标准PQ最小化量化误差→但检索关心的是排序正确性而非绝对距离
   - ScaNN优化量化使得"内积排序"尽可能正确→更适合Maximum Inner Product Search
   - **效果：** 在同等Recall下比HNSW快2-5x→Google内部大量使用

6. **Recall-QPS-Memory三角：**
   | 算法 | Recall | QPS | Memory | 适用 |
   |------|--------|-----|--------|------|
   | HNSW | 极高(>99%) | 高 | 高 | 中等规模,高精度 |
   | IVF-Flat | 高(95%+) | 中 | 中 | 大规模,平衡 |
   | IVF-PQ | 中(90%+) | 高 | 极低 | 超大规模,内存受限 |
   | ScaNN | 高 | 极高 | 中 | 超高QPS需求 |

7. **向量数据库选型：**
   - **Faiss（Meta）：** 库（非数据库）→算法全面→适合研究和自建→无分布式
   - **Milvus：** 分布式→亿级向量→云原生→功能最全→运维复杂
   - **Qdrant：** Rust实现→高性能→过滤+向量混合查询→中等规模首选
   - **Pinecone：** 全托管→零运维→按需扩展→贵→适合快速上线
   - **Chroma：** 轻量→本地开发→适合原型和小项目
   - **Weaviate：** GraphQL接口→多模态→hybrid search内置
   - **选择原则：** 数据量<100万→Chroma/Qdrant；百万-亿级→Milvus/Pinecone；不想运维→Pinecone

### Q2 (11:00) - 为什么Decoder-only架构胜出（vs Encoder-only/Encoder-Decoder）
**题目：** Transformer有三种架构变体——为什么现代LLM几乎全部采用Decoder-only？Encoder-only（BERT）、Encoder-Decoder（T5）和Decoder-only（GPT）的核心区别？Decoder-only在什么条件下超越了其他架构？Causal Language Modeling为什么是更好的预训练目标？Prefix LM是什么折中方案？

**参考答案：**

1. **三种架构回顾：**
   - **Encoder-only（BERT）：**
     - 双向注意力→每个token可以看到所有其他token
     - 预训练：Masked Language Modeling（MLM）→预测被mask的15% token
     - 适合：分类、NER、句子相似度等理解任务
     - 不适合：生成任务（不能自回归生成）
   - **Encoder-Decoder（T5/BART）：**
     - Encoder双向处理输入→Decoder自回归生成输出
     - 预训练：Span Corruption（T5）或Denoising（BART）
     - 适合：翻译、摘要等sequence-to-sequence任务
   - **Decoder-only（GPT）：**
     - 因果注意力→每个token只能看到前面的token
     - 预训练：Causal Language Modeling（CLM）→预测下一个token
     - 适合：通用生成→但通过Scaling也能做理解任务

2. **Decoder-only胜出的原因：**
   - **简洁统一：** 一种架构+一种训练目标→处理所有任务→不需要为每种任务设计不同架构
   - **Scaling效率：** 研究发现Decoder-only在同等计算量下Scale更高效
     - 原因：CLM的每个token都是训练信号→vs MLM只有15%的token是信号→数据效率更高
   - **In-Context Learning：** Decoder-only通过自回归天然支持few-shot→无需微调
   - **生成能力：** 原生支持文本生成→是最通用的能力（理解可以通过生成实现）
   - **工程简洁：** 只需要一个Transformer stack→vs Encoder-Decoder需要两个→参数效率更高

3. **Causal LM为什么是更好的预训练目标：**
   - **每个token都是监督信号：** 序列长度L→有L-1个预测任务→vs MLM只有0.15L个
   - **天然生成能力：** 训练目标就是生成下一个token→不需要额外适配
   - **Scaling友好：** 简单且一致的目标函数→更容易大规模训练→loss曲线平滑
   - **数据利用率：** 同样数据量→CLM提取更多训练信号→Chinchilla显示数据效率关键
   - **缺点：** 单向注意力→理论上理解能力不如双向→但实践中通过Scaling弥补

4. **BERT为什么"落后"了：**
   - BERT擅长理解→但不能生成→在"通用AI助手"时代受限
   - BERT的MLM目标→微调时需要改任务头→每个任务单独微调→不如In-Context Learning灵活
   - BERT的Scaling效果不如GPT→可能因为MLM的训练信号稀疏
   - **但BERT没有消亡：** 在嵌入、分类、检索等专用任务上仍然是好选择→效率高、部署便宜

5. **Prefix LM（折中方案）：**
   - 输入部分用双向注意力（like Encoder）→输出部分用因果注意力（like Decoder）
   - 例：U-PaLM、部分GLM实现
   - 优势：输入理解更充分→输出仍然自回归
   - 劣势：训练和推理实现稍复杂→不如纯Decoder-only简洁
   - 实践中使用较少→Decoder-only + 长context已经够强

6. **T5/Encoder-Decoder的定位：**
   - 在特定任务上仍有优势（翻译、摘要→天然的输入→输出结构）
   - Flan-T5：多任务指令微调后泛化能力强→中小规模模型的好选择
   - 但大规模（>70B）几乎都选择Decoder-only→更好的Scaling特性

7. **总结——架构选择指南：**
   - 通用LLM/聊天机器人→Decoder-only（GPT/LLaMA/Claude）
   - 嵌入/分类/检索→Encoder-only（BERT/RoBERTa）或Bi-Encoder
   - 特定Seq2Seq任务→Encoder-Decoder（T5/BART）→但规模受限
   - 当前趋势：Decoder-only一统天下→其他架构退居专用场景

### Q3 (12:00) - 预训练数据Pipeline工程（爬取/清洗/去重/配比/质量过滤）
**题目：** 预训练数据的质量直接决定LLM的能力上限——工业级数据Pipeline的完整流程？数据来源有哪些？去重的三个层次（精确/模糊/语义）？质量过滤的方法（规则/分类器/困惑度）？数据配比策略？有毒/偏见内容如何过滤？数据Pipeline的规模化工程挑战？

**参考答案：**

1. **数据来源：**
   - **Web爬虫：** Common Crawl（最大，数百TB）→原始HTML→需要大量清洗
   - **书籍：** Books3/Project Gutenberg→高质量→但版权争议
   - **学术论文：** arXiv/Semantic Scholar→高质量→科学知识
   - **代码：** GitHub/Stack Exchange→提升推理能力
   - **百科：** Wikipedia→高质量事实性内容→但规模小
   - **对话/论坛：** Reddit/StackOverflow→对话能力
   - **合成数据：** LLM生成的高质量文本→越来越重要
   - **配比（LLaMA-3参考）：** Web 80% + 代码 8% + 书籍 4% + 学术 4% + 其他 4%

2. **清洗Pipeline（从原始到可用）：**
   ```
   原始HTML → 文本提取(Trafilatura) → 语言检测(fastText)
     → 规则过滤(长度/乱码/广告) → 质量过滤(分类器/PPL)
     → 去重(精确→模糊→语义) → PII移除 → 有毒内容过滤
     → 最终数据集
   ```
   - 通常清洗后只保留原始数据的10-30%

3. **去重三个层次：**
   - **精确去重：**
     - 文档级：SHA-256 hash→完全相同的文档去除
     - URL去重：同一URL不同时间爬取的重复版本
   - **模糊去重（MinHash LSH）：**
     - 将文档分成n-gram（如5-gram）→MinHash生成签名→LSH分桶→相似度>阈值（如0.8）的去除
     - 可以检测"几乎相同"的文档（如转载/略微修改）
     - 规模化：MapReduce/Spark处理→数十亿文档级别
   - **语义去重：**
     - 用embedding模型编码→余弦相似度高的文档聚类→只保留每簇代表
     - 更准确但计算成本高→通常只对子集做
   - **为什么去重重要：** 重复数据导致模型记忆而非理解→训练效率下降→生成质量下降

4. **质量过滤方法：**
   - **规则过滤（Heuristics）：**
     - 最短长度（<50字丢弃）、最长长度（>100K字丢弃）
     - 特殊字符比例（>30%→可能是乱码）
     - 大写字母比例、平均词长、标点符号密度
     - "脏词"列表过滤
   - **分类器过滤：**
     - 训练二分类器（高质量 vs 低质量）→如用Wikipedia(高质量)/随机网页(低质量)训练
     - fastText分类器：轻量高效→可处理数十亿文档
     - GPT-based质量评分：更准确→但成本高→抽样使用
   - **困惑度过滤：**
     - 用小型语言模型（如KenLM）计算每个文档的困惑度
     - PPL太高→可能是乱码/非自然语言
     - PPL太低→可能是重复/模板生成的内容
     - 保留PPL在合理范围内的文档

5. **有毒/偏见内容过滤：**
   - **NSFW分类器：** 过滤色情/暴力文本
   - **毒性分类器：** Perspective API / 自训练的毒性检测器
   - **PII（个人身份信息）移除：**
     - 正则匹配：邮箱/电话/身份证号/银行卡
     - NER模型：人名/地址→替换为占位符
   - **偏见检测：** 检测性别/种族/宗教偏见内容→降采样或移除
   - **权衡：** 过度过滤→模型知识缺失；过少过滤→模型学到有害内容

6. **规模化工程挑战：**
   - **存储：** Common Crawl原始数据~PB级→清洗后~TB级→需要分布式存储
   - **计算：** 去重/过滤需要数千CPU小时→Spark/Ray分布式处理
   - **Reproducibility：** 数据版本管理→记录每步的过滤参数→可重现
   - **增量更新：** 新的Web crawl→增量去重（不是全量重新处理）
   - **质量监控：** 随机抽样人工审查→确保Pipeline没有引入问题
   - **预算：** 数据准备可占预训练总成本的10-20%→不可忽视

### Q4 (13:00) - 学习率Schedule与Batch Size缩放（Warmup/Cosine/WSD/线性缩放）
**题目：** 学习率调度为什么对LLM训练至关重要？Warmup的原理和必要性？Cosine Annealing为什么成为LLM标配？WSD（Warmup-Stable-Decay）Schedule的新趋势？Batch Size和学习率的线性缩放规则？学习率的选择和调优策略？

**参考答案：**

1. **学习率调度的重要性：**
   - 学习率太大→训练不稳定→Loss Spike/发散
   - 学习率太小→训练太慢→在有限计算预算内无法收敛
   - 不同训练阶段需要不同的学习率→初期探索需要大步长→后期精细调整需要小步长
   - LLM训练通常只有一个epoch→没有机会"重来"→调度策略必须一次到位

2. **Warmup（预热）：**
   - **原理：** 训练最初从很小的学习率（如1e-7）线性增加到峰值（如3e-4）
   - **为什么必要：**
     - 训练初期参数随机→梯度方向不可靠→大学习率→可能立即发散
     - Adam优化器需要几百步积累准确的一阶/二阶矩估计→初期估计不准→大LR危险
     - 特别是Post-LN架构→没有Warmup很容易崩溃（Pre-LN对此更鲁棒）
   - **步数：** 通常0.1-1%的总训练步数（如总训练100K步→warmup 1000步）
   - **LLaMA-3：** 8000步warmup（总训练~数百万步）

3. **Cosine Annealing（余弦退火）：**
   - **公式：** lr_t = lr_min + 0.5 × (lr_max - lr_min) × (1 + cos(πt/T))
   - **形状：** 从峰值平滑下降→在末期接近0→先慢后快再慢的S形下降
   - **为什么好：**
     - 平滑下降→不像Step Schedule那样突变→训练更稳定
     - 末期学习率很低→允许模型在loss landscape的底部精细调整
     - 经验上效果一致性好→不太需要调超参
   - **几乎所有现代LLM都用：** GPT-3/4、LLaMA、Mistral、Qwen等
   - **最终学习率：** 通常设为峰值的1-10%（如峰值3e-4→最终3e-5）

4. **WSD Schedule（Warmup-Stable-Decay, MiniCPM/DeepSeek）：**
   - **三阶段：** Warmup → 恒定学习率（Stable）→ 快速衰减（Decay）
   - **动机：** Cosine Schedule需要预先确定总训练步数→但有时想训练更多步→不好调整
   - **WSD优势：**
     - Stable阶段学习率恒定→可以任意延长训练→不需要重新调整schedule
     - 想停的时候加一段Decay就行→灵活性极高
     - 实验表明效果和Cosine Schedule comparable
   - **Decay形式：** 通常是线性下降或余弦下降→持续1-10%的总步数
   - **趋势：** DeepSeek-V3等使用→越来越受欢迎

5. **Batch Size和学习率的缩放关系：**
   - **线性缩放规则（Goyal et al., 2017）：**
     - Batch Size增大k倍→学习率也增大k倍
     - 原理：大Batch的梯度估计更准确→可以用更大步长
   - **平方根缩放：** 有些研究建议lr ∝ √batch_size→更保守但更安全
   - **实践：**
     - Batch Size从256增到4096（16x）→学习率从1.5e-4增到2.4e-3（≈16x）
     - 但需要更长的Warmup（大batch初期不稳定）
   - **关键指标：** 关注的是effective batch size = micro_batch × accumulation × num_gpus
   - **DeepSeek-V3实践：** Batch Size从小逐步增大→训练初期用小batch探索→后期用大batch加速

6. **学习率选择和调优策略：**
   - **峰值学习率范围：** 大模型通常1e-4到5e-4
     - 越大的模型→通常需要越小的学习率（因为梯度scale不同）
     - LLaMA-7B: 3e-4; LLaMA-65B: 1.5e-4
   - **μP（Maximal Update Parameterization）：**
     - 超参数可以从小模型直接迁移到大模型→不需要重新搜索
     - 原理：调整初始化和学习率使得不同宽度的模型有相同的更新幅度
     - CerebrasGPT验证了μP的有效性
   - **实用建议：**
     - 先在小模型上搜索最优学习率→按缩放规则迁移到大模型
     - 如果训练出现Loss Spike→降低学习率是第一选择
     - 监控梯度范数→如果持续增大→学习率可能太高

### Q5 (14:00) - 语音+LLM（Whisper/语音Agent/Speech-to-Speech/GPT-4o Voice）
**题目：** 语音AI的最新进展？Whisper的架构和训练方法？传统语音Pipeline（ASR→LLM→TTS）vs 端到端语音LLM的区别？GPT-4o Voice Mode如何实现低延迟对话？语音Agent的挑战（延迟/打断/情感）？多模态语音理解（语气/情感/说话人识别）？

**参考答案：**

1. **Whisper（OpenAI, 2022）：**
   - **架构：** Encoder-Decoder Transformer
     - Encoder：处理Mel频谱图（音频→80维Mel特征→30秒窗口）
     - Decoder：自回归生成文本token（包括语言标识/时间戳/任务标识）
   - **训练：** 680K小时多语言弱监督数据→Web上的音频+字幕对
   - **多任务：** 语音识别/语音翻译/语言检测/时间戳对齐→通过特殊token控制
   - **优势：** 鲁棒性极强（噪音/口音/多语言）、无需微调开箱即用
   - **模型大小：** Tiny(39M)→Small(244M)→Medium(769M)→Large-v3(1.5B)
   - **局限：** 30秒窗口限制、实时性不足（需要VAD切分）

2. **传统语音Pipeline：**
   ```
   用户说话 → VAD(语音活动检测) → ASR(语音→文字) → LLM(理解+生成) → TTS(文字→语音) → 播放
   ```
   - **延迟累积：** VAD(300ms) + ASR(500ms-2s) + LLM(500ms-2s) + TTS(300ms-1s) = 1.5-5s
   - **信息损失：** ASR只输出文字→丢失语气/情感/语速/停顿等副语言信息
   - **打断处理困难：** 需要持续监听+判断是否打断→pipeline复杂

3. **端到端语音LLM（GPT-4o Voice/Gemini Live）：**
   - **核心思路：** LLM直接处理音频token→不需要ASR/TTS中间步骤
   - **架构：**
     - 音频编码器将语音转为离散音频token（如使用Neural Audio Codec→Encodec/SoundStream）
     - LLM同时处理文本token和音频token→多模态
     - 输出也是音频token→直接解码为语音→保留语气/情感
   - **延迟优势：** 省去ASR+TTS→端到端延迟<500ms→接近实时对话
   - **信息保留：** 音频token包含语气/情感/语速→LLM可以理解和生成富有表情的语音

4. **语音Agent的核心挑战：**
   - **延迟：** 人类对话中停顿>600ms感觉不自然→系统端到端必须<500ms
   - **打断处理（Barge-in）：**
     - 用户随时可能打断→Agent需要立即停止播放→开始听
     - 需要持续的VAD监听→即使在播放回复时
     - 半双工→全双工（同时听说）是未来方向
   - **轮次检测（Turn-taking）：**
     - 用户说完了吗？停顿≠结束→可能在思考
     - 需要语义+韵律+停顿时长综合判断
   - **情感/语气：**
     - 用户生气→Agent应该更empathetic
     - 传统ASR+LLM pipeline丢失这些信号
   - **多语言/口音/噪音：** 真实环境音频质量差→鲁棒性要求高

5. **语音Agent架构选择：**
   - **方案A（Pipeline）：** Whisper + LLM + TTS（如ElevenLabs/Azure TTS）
     - 优点：每个组件可独立优化、成熟可靠
     - 缺点：延迟高、信息损失
     - 适合：对延迟不敏感的语音助手（如语音笔记）
   - **方案B（端到端）：** 原生多模态模型（GPT-4o/Gemini）
     - 优点：低延迟、信息完整、自然
     - 缺点：模型闭源/成本高、定制难
     - 适合：实时对话产品
   - **方案C（混合）：** 快速ASR（Streaming Whisper）+ LLM + 快速TTS
     - Streaming ASR：边听边转写→不等说完→减少延迟
     - 快速TTS：生成第一个音素就开始播放→流式输出
     - 延迟可控到800ms-1.2s→实用折中

6. **前沿方向：**
   - **全双工对话：** 同时听+说→像人类对话→Moshi（Kyutai）
   - **语音克隆：** 用少量样本克隆说话人声音→个性化Agent
   - **音频理解：** 不只是语音→理解环境音/音乐/音效→多模态
   - **实时翻译：** 说中文→直接输出英文语音→保留原始语气

### Q6 (15:00) - LLM语义缓存（Exact Cache/Semantic Cache/GPTCache/成本优化）
**题目：** LLM API成本高——如何通过缓存减少重复调用？精确缓存和语义缓存的区别？语义缓存的实现原理（Embedding相似度匹配）？GPTCache框架的架构？缓存命中率和回答质量的trade-off？缓存失效和更新策略？在生产环境中的实际效果？

**参考答案：**

1. **LLM缓存的动机：**
   - LLM API按token计费→相似问题重复调用=浪费钱
   - 实际场景中大量重复/相似查询：客服FAQ、代码补全、搜索补全
   - 缓存还能减少延迟→缓存命中→毫秒级返回（vs LLM推理的数秒）
   - 潜在节省：30-70%的API调用成本（取决于查询重复率）

2. **精确缓存（Exact Cache）：**
   - **原理：** prompt完全一致→直接返回缓存结果
   - **实现：** hash(prompt) → Redis/内存→O(1)查找
   - **优点：** 实现极简、零误差、速度快
   - **缺点：** 命中率低→"今天天气怎么样"和"今天天气如何"就无法匹配
   - **适用：** API代理层的基础缓存→必须有→成本极低

3. **语义缓存（Semantic Cache）：**
   - **原理：** 将query编码为embedding→在缓存中找语义最相似的query→相似度>阈值则命中
   - **流程：**
     ① 新query → embedding model → query_vector
     ② 在向量数据库中检索最相似的cached_query
     ③ if similarity > threshold (如0.95)：返回缓存的response
     ④ else：调用LLM → 存入缓存
   - **优势：** "今天天气怎么样"和"今天的天气情况"可以命中→大幅提升命中率
   - **挑战：** 相似度阈值如何设定→太低则误匹配→太高则命中率低

4. **GPTCache框架架构：**
   - **组件：**
     - **Embedding Manager：** 将query编码为向量（支持OpenAI/Sentence-Transformers/Cohere）
     - **Cache Store：** 存储query-response对（SQLite/Redis/DynamoDB）
     - **Vector Store：** 存储query向量用于相似度检索（Faiss/Milvus/Qdrant）
     - **Similarity Evaluator：** 判断是否足够相似（余弦相似度/LLM-based判断）
     - **Eviction Policy：** 缓存淘汰（LRU/TTL/容量限制）
   - **两级评估（可选）：** 先embedding相似度粗筛→再用小LLM精判是否可以复用

5. **缓存命中率 vs 回答质量的trade-off：**
   - **阈值过低（如0.85）：**
     - 命中率高→成本节省多
     - 但可能返回不相关的缓存结果→回答质量下降
     - 例："苹果公司股价"匹配到"苹果的营养价值"→灾难
   - **阈值过高（如0.99）：**
     - 几乎只匹配完全相同的查询→命中率低→节省有限
     - 但回答质量有保证
   - **实践：** 从0.95开始→监控用户反馈→如果投诉增加→提高阈值
   - **分场景阈值：** FAQ类查询阈值可以低一些→创意/个性化查询阈值要高

6. **缓存失效和更新策略：**
   - **TTL（Time-To-Live）：** 缓存条目设置过期时间→时效性内容（天气/新闻）TTL短
   - **LRU淘汰：** 最近最少使用的条目被淘汰→控制缓存大小
   - **主动失效：** 知识库更新→相关缓存条目失效→强制LLM重新生成
   - **版本管理：** 模型版本变更→旧缓存可能不适用→清空或逐步替换
   - **注意：** 缓存的system prompt/temperature等参数也要匹配→不同参数不能共享缓存

7. **生产环境效果（实际数据）：**
   - **客服场景：** 缓存命中率40-60%→成本降低35-50%→延迟从2s降到50ms
   - **代码补全：** 命中率20-30%→常见pattern重复率高
   - **搜索增强：** 命中率10-20%→查询多样性高→但每次节省的成本大（长prompt）
   - **总体ROI：** embedding计算+向量存储的成本 << 节省的LLM API成本→通常1周回本

### Q7 (16:00) - RETRO / 检索增强预训练（Retrieval-Augmented Pre-training）
**题目：** 除了在推理时使用RAG，能否在预训练阶段就引入检索？RETRO（Retrieval-Enhanced Transformer）的原理？和标准RAG（推理时检索）的本质区别？预训练时检索的优势和局限？RETRO++和Atlas等后续改进？这条路线为什么没有成为主流？

**参考答案：**

1. **动机——预训练时检索：**
   - 标准LLM：所有知识都必须编码在参数中→模型越大知识越多→但参数效率低
   - RETRO思路：不用把所有知识存参数→训练时从外部知识库检索→让模型学会"查资料"
   - 类似：开卷考试（可以查书）vs 闭卷考试（全靠记忆）→开卷需要更少的记忆

2. **RETRO（Borgeaud et al., 2022, DeepMind）：**
   - **架构：**
     - 将输入序列分成chunk（如64个token一块）
     - 对每个chunk→从预构建的万亿token数据库中检索最近邻chunk
     - 用Cross-Attention将检索到的chunk信息融入模型
   - **检索数据库：** 预训练语料的所有chunk→编码为向量→构建FAISS索引→2T tokens
   - **Cross-Attention融合：** 在Transformer的特定层（每3层一次）加入Chunked Cross-Attention
     - 输入chunk的表示作为Query→检索到的chunk作为Key/Value
   - **关键结果：** RETRO 7.5B ≈ 标准Transformer 25B（在perplexity上）→参数效率提升3x+

3. **和推理时RAG的本质区别：**
   | 维度 | RETRO（预训练时检索） | 标准RAG（推理时检索） |
   |------|----------------------|---------------------|
   | 训练 | 模型学会利用检索结果 | 模型不知道有检索→直接拼接 |
   | 架构 | 需要修改Transformer（加Cross-Attention） | 不改架构→检索结果放context |
   | 数据库 | 预训练语料的万亿token | 任意外部知识库 |
   | 更新知识 | 更新数据库即可→不需要重新训练 | 更新知识库即可 |
   | 成本 | 训练时需要检索→增加训练复杂度 | 推理时需要检索→增加延迟 |

4. **优势和局限：**
   - **优势：**
     - 参数效率高→小模型也能有大模型的知识量
     - 知识可更新→换数据库就行→不需要重新训练
     - 减少幻觉→有外部知识支撑
   - **局限：**
     - 训练基础设施复杂→需要在训练时维护大规模向量检索系统
     - 检索延迟增加训练时间
     - 检索数据库的质量和覆盖度是瓶颈
     - 模型对检索结果产生依赖→数据库不可用时性能严重下降

5. **后续改进：**
   - **RETRO++（2023）：** 改进检索策略+预训练后继续训练（continual pre-training with retrieval）
   - **Atlas（Izacard et al., 2023）：** Few-shot学习+检索增强→T5基础→在知识密集任务上SOTA
   - **REALM（Google, 2020）：** 更早的检索增强预训练→端到端训练检索器和生成器
   - **Toolformer（2023）：** 教模型使用工具（包括搜索引擎）→更灵活的检索方式

6. **为什么没有成为主流：**
   - **工程复杂度：** 训练时需要维护TB级向量索引+每步检索→极大增加训练pipeline复杂度
   - **Scaling问题：** 模型本身的Scaling已经足够好→增加参数比增加检索更简单
   - **推理时RAG更灵活：** 不需要改架构→任何模型都能用→解耦更好
   - **趋势：** 大力出奇迹（Scaling Laws）路线暂时胜出→但检索增强思路在特定场景仍有价值
   - **未来可能：** 当Scaling遇到瓶颈→检索增强预训练可能回归→尤其是知识密集型场景

### Q9 (18:00) - Transformer权重初始化与训练稳定性（Xavier/He/μP/Depth Scaling）
**题目：** 权重初始化为什么对Transformer训练至关重要？Xavier和He初始化的数学原理？为什么Transformer需要特殊的初始化策略？GPT-2的残差缩放（1/√N）是什么？μP（Maximal Update Parameterization）如何解决超参数迁移？Depth Scaling的原理？

**参考答案：**

1. **为什么初始化重要：**
   - 不好的初始化→前向传播信号爆炸/消失→反向传播梯度爆炸/消失→训练失败
   - 目标：保持每层的激活值方差稳定（不放大也不缩小）→信号在深层网络中"平稳"传递
   - 大模型（100+层）对初始化更敏感→初始值的微小偏差经过100层累积可能导致灾难

2. **Xavier初始化（Glorot, 2010）：**
   - **适用：** 线性层 + Sigmoid/Tanh激活
   - **公式：** W ~ U(-√(6/(fan_in + fan_out)), √(6/(fan_in + fan_out)))
   - **原理：** 使得每层输出的方差 = 输入的方差→前向传播方差稳定
   - 推导基于线性假设→Sigmoid/Tanh在0附近近似线性→成立

3. **He初始化（He, 2015）：**
   - **适用：** ReLU/GELU等非对称激活
   - **公式：** W ~ N(0, 2/fan_in)
   - **原理：** ReLU会"砍掉"一半的值（负值→0）→方差减半→需要2倍补偿
   - 大多数Transformer的FFN用GELU/SwiGLU→He初始化更合适

4. **GPT-2的残差缩放（1/√N）：**
   - **问题：** 深层Transformer有很多残差连接→每层加一次残差→方差不断增长
   - **解决：** 残差路径上的最后一层线性层的初始权重缩放为 1/√(2N)
     - N = Transformer层数
     - 2 = 每层有两个残差连接（Attention + FFN）
   - **效果：** 深层网络的激活值方差保持稳定→训练更稳定
   - 几乎所有GPT系列模型都使用这个trick

5. **μP（Maximal Update Parameterization, Yang et al., 2022）：**
   - **核心问题：** 在小模型上找到的最优超参数（学习率/初始化）→直接用在大模型上效果不好
   - **原因：** 标准参数化下，模型宽度变化→每层更新的"幅度"也变化→超参数不可迁移
   - **μP的解决：**
     - 调整初始化和学习率的缩放规则→使得不同宽度模型的参数更新幅度一致
     - 具体：某些层的LR ∝ 1/width，初始化 ∝ 1/√width
   - **实践价值：** 在128维宽度的小模型上搜索超参→直接应用到4096维的大模型→效果一致
   - **节省：** 超参搜索成本从$1M+降到$1K→巨大的工程价值

6. **Depth Scaling：**
   - **问题：** 模型层数增加→即使有残差连接→深层的梯度仍可能不稳定
   - **方法：** 随深度调整各层的初始化/学习率
     - 深层初始化更小→防止深层信号爆炸
     - 或：深层学习率更小→更新更保守
   - **DeepNorm（Microsoft）：** 修改残差连接为 x + α·SubLayer(x)→α随深度调整→支持1000+层
   - **Sub-LN：** 在残差内部加额外的LayerNorm→进一步稳定深层

7. **现代LLM的初始化实践（总结）：**
   - Embedding层：N(0, 0.02) 或 N(0, 1/√d_model)
   - Attention QKV投影：He初始化 或 N(0, 1/√d_model)
   - Attention Output投影：N(0, 1/√(d_model·N_layers))→GPT-2 trick
   - FFN层：He初始化
   - LM Head：通常和Embedding层共享权重（weight tying）

### Q10 (19:00) - LLM评估体系深入（Human Eval/ELO/Chatbot Arena/LMSys方法论）
**题目：** 如何全面评估一个LLM的能力？静态Benchmark（MMLU/GSM8K）的局限？Chatbot Arena的ELO评分系统如何工作？为什么人类偏好评估是"金标准"？自动化评估（LLM-as-Judge）的可靠性？如何构建企业内部的LLM评估体系？

**参考答案：**

1. **静态Benchmark的局限：**
   - **数据污染：** 测试集可能被训练数据包含→分数虚高
   - **饱和效应：** 多模型在MMLU上>85%→失去区分度
   - **不代表真实使用：** MMLU测知识记忆→不测对话/创意/指令跟随能力
   - **格式依赖：** 模型可能擅长选择题格式但不擅长开放回答
   - **仍然有价值：** 作为基础能力的"体检"→但不能作为唯一评估

2. **Chatbot Arena / LMSYS方法论：**
   - **流程：**
     ① 用户在平台上输入问题
     ② 系统随机选择两个匿名模型分别回答
     ③ 用户选择更好的回答（或平局）
     ④ 基于所有投票结果计算ELO评分
   - **ELO评分系统（借鉴国际象棋）：**
     - 每个模型有一个ELO分数（初始1000）
     - 强模型赢弱模型→两者分数变化小；弱模型赢强模型→变化大
     - P(A赢B) = 1 / (1 + 10^((R_B - R_A)/400))
     - 经过数万次投票→分数稳定→排名可靠
   - **为什么是"金标准"：**
     - 盲测→消除品牌偏见
     - 真实用户真实问题→代表实际使用场景
     - 持续更新→不会饱和

3. **人类评估的挑战：**
   - **成本高：** 每次对比需要人工→大规模评估昂贵
   - **一致性：** 不同人偏好不同→Inter-Annotator Agreement需要关注
   - **偏见：**
     - 长度偏见：倾向选更长的回答
     - 格式偏见：偏好有markdown/列表的回答
     - 顺序偏见：倾向选第一个呈现的答案
   - **缓解：** 多人评估取多数→随机化呈现顺序→控制回答长度

4. **LLM-as-Judge（自动化评估）：**
   - **方法：** 用强LLM（GPT-4/Claude）评估其他模型的输出
   - **模式：**
     - Pointwise：对单个输出打1-5分
     - Pairwise：比较两个输出哪个更好
     - Reference-based：与标准答案对比
   - **可靠性：**
     - 和人类评估的一致性：70-85%（取决于任务和评估prompt设计）
     - 已知偏差：position bias、verbosity bias、self-enhancement bias
     - 缓解：swap位置各评一次→取平均；多个Judge模型投票
   - **适用：** 大规模自动化评估→筛选→人工只评审边界case

5. **多维度评估框架：**
   - **基础能力：** MMLU（知识）、GSM8K/MATH（数学）、HumanEval（代码）、BBH（推理）
   - **指令跟随：** IFEval（严格指令遵从）、MT-Bench（多轮对话质量）
   - **长文本：** Needle-in-Haystack、RULER、LongBench
   - **安全：** TruthfulQA、BBQ（偏见）、Red-team测试
   - **Agent能力：** SWE-bench、GAIA、WebArena
   - **人类偏好：** Chatbot Arena ELO、AlpacaEval

6. **企业内部评估体系构建：**
   - **第一步：收集真实case**
     - 从业务场景收集100-500个代表性query
     - 覆盖不同难度/类型/领域
     - 包含"expected behavior"标注（不一定是标准答案→但定义什么是好回答）
   - **第二步：自动化评估pipeline**
     - 对每个case运行被评估模型→收集输出
     - LLM-as-Judge打分→多维度（准确性/完整性/安全性/格式）
     - 自动生成评估报告
   - **第三步：人工校准**
     - 定期抽样→人工审核LLM Judge的评分→校准prompt
     - 对LLM Judge和人工不一致的case重点分析
   - **第四步：回归测试**
     - 每次模型/prompt更新→自动运行评估→分数下降→阻止上线
   - **关键：** 评估集是活的→持续添加新case（尤其是线上bad case）

### Q11 (20:00) - Fine-tuning策略全景对比（Full/LoRA/Prefix-Tuning/Adapter/IA³）
**题目：** 除了LoRA，还有哪些参数高效微调（PEFT）方法？Prefix-Tuning的虚拟token原理？Adapter Layer的插入方式？IA³（Infused Adapter by Inhibiting and Amplifying Inner Activations）的门控机制？各方法在参数量、效果、推理开销上的对比？什么场景选什么方法？

**参考答案：**

1. **全量微调（Full Fine-tuning）：**
   - 更新模型所有参数→最高质量但最贵
   - 显存需求：模型参数(2bytes) + 梯度(2bytes) + 优化器状态(8bytes) ≈ 12bytes/param
   - 70B模型→需要~840GB→至少8×A100
   - 适用：有充足算力 + 任务和预训练分布差距大 + 追求极致质量

2. **LoRA回顾（基准线）：**
   - ΔW = BA→可训练参数 = (d+k)×r→通常占原参数的0.1-1%
   - 推理时合并→零额外开销
   - 效果：简单任务接近全量→复杂任务差2-5%
   - 当前PEFT的默认选择

3. **Prefix-Tuning（Li & Liang, 2021）：**
   - **原理：** 在每层的输入前拼接可学习的"虚拟token"（prefix）
   - prefix是直接优化的连续向量→不对应任何真实token
   - 每层有独立的prefix→共同组成一个"软提示"
   - **参数量：** prefix_length × num_layers × hidden_dim→通常prefix 10-100个→参数极少
   - **优点：** 参数最少→适合极端资源受限
   - **缺点：**
     - 占用context window→prefix的token位置不能给真实输入用
     - 效果通常略逊于LoRA→尤其在生成任务上
     - 训练不稳定→需要精心调整
   - **P-Tuning v2：** 类似Prefix-Tuning但只在某些层加prefix→更灵活

4. **Adapter Layers（Houlsby et al., 2019）：**
   - **原理：** 在Transformer每层中插入小型瓶颈网络
   - 结构：x → LayerNorm → Down-project(d→r) → 非线性 → Up-project(r→d) → 残差连接
   - 插入位置：Attention之后 + FFN之后（每层两个Adapter）
   - **参数量：** 2 × num_layers × 2 × (d×r + r×d) → 比LoRA多（每层两个Adapter各两个投影）
   - **推理开销：** 有额外计算→不像LoRA可以合并→推理时有开销
   - **优点：** 早期PEFT先驱→研究完善
   - **缺点：** 推理有额外延迟→LoRA出来后使用减少

5. **IA³（Few-shot PEFT, Liu et al., 2022）：**
   - **原理：** 学习三个缩放向量（l_k, l_v, l_ff）→分别缩放K、V和FFN的中间激活
   - 极其简单：只是逐元素乘缩放因子→不改变架构
   - **参数量：** 3 × num_layers × hidden_dim→比LoRA更少
   - **效果：** 在few-shot场景下效果好→但在大规模微调中不如LoRA
   - **优点：** 参数极少→适合同时维护大量任务的Adapter
   - **缺点：** 表达能力有限→复杂适配不够

6. **全面对比：**
   | 方法 | 可训练参数 | 推理开销 | 效果(vs Full) | 适用场景 |
   |------|-----------|---------|--------------|---------|
   | Full FT | 100% | 无 | 100% | 算力充足,最高质量 |
   | LoRA | 0.1-1% | 无(合并后) | 95-100% | 默认PEFT选择 |
   | QLoRA | 0.1-1% | 无 | 93-98% | 单GPU微调大模型 |
   | DoRA | 0.1-1% | 无 | 96-101% | 防遗忘+高质量 |
   | Prefix-Tuning | 0.01-0.1% | 占context | 88-95% | 极端参数限制 |
   | Adapter | 1-5% | 有(额外层) | 93-98% | 研究/遗留系统 |
   | IA³ | 0.01% | 极小 | 85-93% | 大量任务切换 |

7. **选择指南：**
   - **默认→LoRA：** 最佳平衡（效果好+推理零开销+生态完善）
   - **单GPU大模型→QLoRA：** 4-bit量化+LoRA→民主化微调
   - **防遗忘→DoRA：** 权重分解→安全性更高
   - **数十到数百个任务→LoRA+S-LoRA serving：** 共享base+动态Adapter
   - **研究/实验→Prefix-Tuning/IA³：** 参数极少→快速实验

### Q12 (21:00) - LLM结构化剪枝（Structured Pruning/SparseGPT/Wanda/LLM-Pruner）
**题目：** 除了量化，剪枝（Pruning）是另一种模型压缩方法——LLM剪枝的主要方法？非结构化 vs 结构化剪枝的区别？SparseGPT如何一次性剪枝到50%稀疏度？Wanda为什么更简单更快？LLM-Pruner的结构化剪枝策略？剪枝 vs 量化 vs 蒸馏的对比？

**参考答案：**

1. **剪枝的基本概念：**
   - 将模型中"不重要"的参数设为0（或删除）→减少模型大小和计算量
   - **非结构化剪枝（Unstructured）：** 任意位置的单个权重可以被剪掉→稀疏矩阵
     - 优点：最灵活→可以达到高稀疏度（如90%）
     - 缺点：稀疏矩阵在GPU上加速困难→需要专用硬件/kernel
   - **结构化剪枝（Structured）：** 删除整行/列/注意力头/FFN神经元/整层
     - 优点：剪枝后仍然是密集矩阵→标准硬件可以直接加速
     - 缺点：粒度粗→同等剪枝率下质量损失更大

2. **SparseGPT（Frantar & Alistarh, 2023）：**
   - **核心：** 一次性（one-shot）将LLM剪枝到50%稀疏度→无需重新训练
   - **基于OBS（Optimal Brain Surgeon）思想：**
     - 剪掉一个权重→用Hessian信息补偿其他权重→最小化输出误差
     - 逐列处理（类似GPTQ的逐列量化思路）
   - **效果：** LLaMA-7B/13B在50%稀疏度下→perplexity仅升1-2
   - **组合：** 可以和量化一起用→50%稀疏 + INT4量化→压缩8x+
   - **需要校准数据：** ~128条→计算Hessian

3. **Wanda（Pruning by Weights and Activations, Sun et al., 2023）：**
   - **核心创新：极其简单→不需要Hessian，不需要迭代**
   - **重要性度量：** |W_ij| × ||X_j||₂（权重绝对值 × 对应输入激活的L2范数）
   - **直觉：** 权重小 AND 对应激活也小→说明真的不重要→剪掉
   - **流程：** 一次前向传播收集激活统计→按重要性排序→剪掉最不重要的→完毕
   - **效果：** 和SparseGPT comparable→但快10-100倍（无需Hessian计算）
   - **适合大模型：** 简单快速→可以对100B+模型做剪枝

4. **LLM-Pruner（Ma et al., 2023）：**
   - **结构化剪枝→直接删除神经元/头/层**
   - **三步：**
     ① 依赖分析：哪些参数组是"一组"必须一起删（如FFN的两层是耦合的）
     ② 重要性评估：基于梯度信息评估每组的重要性
     ③ 剪枝+轻量恢复：删除不重要的组→用少量数据LoRA微调恢复质量
   - **效果：** LLaMA-7B剪掉20%参数→经过LoRA恢复→性能下降5-10%
   - **优势：** 剪枝后模型是标准dense模型→无需稀疏硬件支持

5. **层剪枝（Layer Pruning）：**
   - 最极端的结构化剪枝→直接删除整层
   - 研究发现：Transformer的中间层很多是冗余的→删除后影响surprisingly小
   - **ShortGPT（2024）：** 用Block Influence（BI）度量每层重要性→删除BI最低的层
   - 效果：删除25%的层→在某些任务上性能下降<5%
   - **Depth Pruning + Width Pruning组合→同时减少层数和每层宽度**

6. **剪枝 vs 量化 vs 蒸馏：**
   | 维度 | 剪枝 | 量化 | 蒸馏 |
   |------|------|------|------|
   | 压缩方式 | 删除参数 | 降低精度 | 训练小模型 |
   | 压缩率 | 2-5x | 2-4x | 任意 |
   | 质量损失 | 中等 | 小 | 取决于Student |
   | 需要训练 | 通常不需要(one-shot) | 不需要(PTQ) | 需要 |
   | 硬件要求 | 稀疏需要特殊支持 | 标准(INT8/INT4支持好) | 标准 |
   | 可组合 | ✅ | ✅ | ✅ |
   - **实际优先级：** 量化（最简单最有效）> 蒸馏（质量最可控）> 剪枝（额外收益）
   - **最佳组合：** 结构化剪枝 + INT4量化→双重压缩→8-16x总压缩率

### Q13 (22:00) - 数据飞轮与持续学习（Data Flywheel/Online Learning/Continual Alignment）
**题目：** LLM上线后如何持续改进？数据飞轮（Data Flywheel）的运转机制？如何从用户反馈中自动构造训练数据？Continual Alignment（持续对齐）的挑战？Online Learning在LLM中是否可行？模型版本管理和A/B测试的最佳实践？

**参考答案：**

1. **数据飞轮（Data Flywheel）：**
   - **核心循环：**
     ```
     模型上线 → 用户使用 → 收集反馈数据 → 清洗标注 → 训练改进 → 模型更新 → 更好的用户体验 → 更多用户 → 更多数据 → ...
     ```
   - **飞轮加速：** 用户越多→数据越多→模型越好→用户更多→自我加速
   - **关键：** 如何高效地将用户反馈转化为训练信号→这是飞轮的"轴承"

2. **从用户反馈构造训练数据：**
   - **显式反馈：**
     - 👍/👎按钮→直接获得偏好信号→可用于DPO训练
     - 用户编辑/修改模型输出→edited version = chosen, original = rejected
     - 用户重新提问（rephrase）→说明第一次回答不满意
   - **隐式反馈：**
     - 用户是否继续对话→继续=满意→离开=可能不满意
     - 复制/分享行为→高质量回答的正信号
     - 对话长度/深度→深度交互暗示价值
   - **自动标注：**
     - 用LLM-as-Judge自动评分→筛选高分回答作为SFT数据
     - 对比不同版本模型的输出→自动构造偏好对
   - **质量控制：** 自动数据需要抽样人工审核→防止bias放大

3. **Continual Alignment（持续对齐）的挑战：**
   - **灾难性遗忘：** 用新数据对齐→可能忘记旧的对齐→安全能力退化
   - **分布漂移：** 用户使用模式变化→旧模型的假设不再成立
   - **偏好漂移：** 社会规范变化→之前"安全"的回答现在可能不合适
   - **解决方案：**
     - Replay：新数据中混入20-30%的旧对齐数据→防止遗忘
     - EWC（Elastic Weight Consolidation）：对重要参数施加正则化→限制大幅更新
     - 频率控制：不要每天更新→积累足够数据后batch更新（如每月/每季度）

4. **Online Learning在LLM中的可行性：**
   - **传统Online Learning：** 每条新数据立即更新模型→实时适应
   - **LLM现实：** 不可行
     - 单次梯度更新太不稳定→LLM需要大量数据的平均梯度
     - 灾难性遗忘风险极高
     - 训练基础设施不支持实时更新
   - **可行的近似：**
     - **Micro-batch更新：** 积累1000-10000条新数据→做一轮轻量微调
     - **RAG作为"实时学习"：** 新知识立即放入知识库→无需更新模型
     - **Prompt缓存更新：** 更新system prompt/few-shot examples→快速适应
     - **LoRA热切换：** 训练新LoRA→动态切换→秒级"更新"

5. **模型版本管理和A/B测试：**
   - **版本管理：**
     - 每个模型版本：base_model_id + fine_tune_version + adapter_version
     - 存储：模型权重在S3/GCS + 元数据在数据库 + 评估结果在评估系统
     - 回滚能力：随时切回任何历史版本→5分钟内完成
   - **A/B测试：**
     - 新模型先5%流量→监控关键指标（满意度/错误率/成本）→逐步放量
     - 统计显著性：至少1000-5000次对比→bootstrap confidence interval
     - 多指标平衡：质量提升但成本增加→需要综合ROI决策
   - **Shadow Mode：** 新模型并行推理但不展示给用户→只做评估→零风险验证

6. **完整的持续改进流程：**
   ```
   Week 1-2: 收集用户反馈 + 自动标注
   Week 3: 数据清洗 + 质量审核
   Week 4: 训练新版本（SFT/DPO）+ 离线评估
   Week 5: Shadow Mode验证
   Week 6: A/B测试（5%→20%→50%）
   Week 7: 全量上线 + 监控
   ↓ 重复 ↓
   ```

### Q14 (23:00) - 快速问答轮：30个核心概念各一句话解释（模拟面试Rapid-fire）
**题目：** 面试尾声常见的"快速问答"环节——面试官连续抛出概念要求简短解释。请对以下30个概念各用一句话精准解释。

**参考答案：**

1. **Attention：** Query和Key计算相似度后对Value加权求和，让模型选择性地关注输入的不同部分。

2. **FlashAttention：** 通过分块计算和在线Softmax避免将L×L注意力矩阵写入HBM，将IO复杂度降低一个量级。

3. **KV Cache：** 推理时缓存已计算的Key和Value，使每步只需计算新token的QKV而非重新计算全部序列。

4. **GQA：** 将注意力头分组，每组共享K/V，在MHA和MQA之间取得质量和效率的平衡。

5. **RoPE：** 通过对Q/K向量施加位置相关的旋转，使注意力分数自然编码相对位置信息。

6. **LoRA：** 冻结原始权重，用两个低秩矩阵BA逼近权重更新ΔW，将微调参数量降低100倍以上。

7. **SFT：** 用高质量的指令-回答对微调预训练模型，教会模型遵循指令和对话格式。

8. **RLHF：** 用人类偏好训练奖励模型，再用PPO算法优化策略模型使其输出更符合人类偏好。

9. **DPO：** 将RLHF的奖励建模和策略优化合为一步，直接在偏好对上训练，无需显式奖励模型。

10. **GRPO：** 对同一prompt采样多个回答，用组内相对排名替代Critic模型估计优势函数。

11. **MoE：** 用路由机制为每个token选择少数专家网络激活，用大参数量小计算量实现高性能。

12. **RAG：** 检索外部知识库的相关文档拼入prompt，让模型基于最新事实生成回答以减少幻觉。

13. **CoT：** 让模型在给出最终答案前展示逐步推理过程，将复杂问题分解为可靠的简单子步骤。

14. **BPE：** 从字符开始反复合并最高频的相邻对构建词表，平衡词表大小和文本编码效率。

15. **PagedAttention：** 将KV Cache按页动态分配和回收，像操作系统虚拟内存一样消除显存碎片。

16. **Continuous Batching：** 在每个生成步检查已完成的请求并立即加入新请求，使GPU始终满负载。

17. **Speculative Decoding：** 用小模型快速猜测多个token再由大模型一次性验证，在不损失质量的前提下加速推理。

18. **GPTQ：** 基于Hessian信息逐列量化权重并补偿剩余列的误差，实现高质量的INT4训练后量化。

19. **Knowledge Distillation：** 用大模型的软标签（含类间关系的暗知识）训练小模型，使其获得超出自身规模的能力。

20. **Catastrophic Forgetting：** 微调新任务时新梯度与旧能力所需梯度方向冲突，导致原有能力退化。

21. **Constitutional AI：** 定义一组显式规则（宪法），让AI根据规则自我批评和修正回答以实现无需人工标注的对齐。

22. **Prefix Caching：** 缓存共享System Prompt的KV Cache供后续请求复用，避免重复的Prefill计算。

23. **Test-Time Compute：** 在推理时投入更多计算（更长推理链/多次采样/树搜索），用推理时间换取更高准确率。

24. **Scaling Laws：** 模型性能随训练计算量/参数量/数据量呈幂律关系提升，可预测最优的资源分配比例。

25. **CLIP：** 用对比学习在共享空间中对齐图像和文本表示，实现零样本跨模态理解。

26. **Reward Hacking：** 模型学会利用奖励模型的漏洞获得高分但实际输出质量低下的对齐失败现象。

27. **Prompt Injection：** 攻击者通过精心构造的输入覆盖系统预设指令，使LLM执行非预期行为。

28. **Hallucination：** 模型生成看似合理但事实上不正确或无法从输入验证的内容。

29. **ZeRO：** 将优化器状态/梯度/参数按GPU分片存储消除数据并行中的冗余，在不增加通信的前提下大幅节省显存。

30. **Semantic Cache：** 用embedding相似度匹配查询，对语义相近的问题直接返回缓存结果以降低LLM调用成本。

### Q15 (00:00) - System Design：实时对话语音AI Agent（类GPT-4o Voice Mode）
**题目：** 设计一个实时对话语音AI Agent系统（类似GPT-4o Voice Mode），支持：①端到端延迟<500ms ②自然的轮次切换和打断处理 ③情感感知和表达 ④多语言支持 ⑤工具调用能力（查天气/订餐/日历等）。从整体架构、音频处理Pipeline、对话管理、工具集成、延迟优化几个方面展开。

**参考答案：**

1. **整体架构：**
   ```
   麦克风 → 音频流
              ↓
   ┌─────────────────────────────────┐
   │  Audio Frontend                  │
   │  VAD + 降噪 + Echo Cancellation │
   └──────────┬──────────────────────┘
              ↓
   ┌─────────────────────────────────┐
   │  理解层 (Understanding)          │
   │  Streaming ASR + 情感检测        │
   │  OR 端到端音频LLM                │
   └──────────┬──────────────────────┘
              ↓
   ┌─────────────────────────────────┐
   │  对话管理 (Dialog Manager)       │
   │  状态机 + 意图识别 + 工具路由    │
   └──────────┬──────────────────────┘
              ↓
   ┌─────────────────────────────────┐
   │  生成层 (Generation)             │
   │  LLM推理 + 流式输出              │
   └──────────┬──────────────────────┘
              ↓
   ┌─────────────────────────────────┐
   │  语音合成 (TTS)                  │
   │  流式合成 + 情感表达              │
   └──────────┬──────────────────────┘
              ↓
   扬声器 ← 音频流
   ```

2. **音频前端处理（Audio Frontend）：**
   - **VAD（Voice Activity Detection）：**
     - 实时检测用户是否在说话→Silero VAD（轻量/精准/16ms帧）
     - 端点检测（Endpointing）：用户停止说话的判定→通常600-800ms静默→但需要结合语义
   - **Echo Cancellation：** Agent说话时用户也在说→需要消除Agent播放的音频回声
   - **降噪：** 环境噪音过滤→RNNoise（实时/轻量）
   - **延迟要求：** 整个前端<50ms→用流式处理（逐帧/每20ms处理一次）

3. **理解层——两种方案：**
   - **方案A（Pipeline）：**
     - Streaming ASR：Whisper-streaming / Conformer-based→边听边转写
     - 不等用户说完→partial transcript实时送给LLM→减少延迟
     - 情感检测：并行的情感分类模型→从音频特征（语调/语速/音量）判断情绪
   - **方案B（端到端）：**
     - 音频token化：Neural Audio Codec（如Encodec）将音频编码为离散token
     - 多模态LLM直接处理音频token→省去ASR步骤
     - 保留完整的副语言信息（语气/停顿/情感）
   - **推荐：** 短期用方案A（成熟可控）→长期迁移到方案B（更自然）

4. **对话管理（Dialog Manager）：**
   - **轮次管理（Turn-taking）：**
     - 不能只靠静默判断→"嗯..."可能是在思考不是结束
     - 多信号融合：静默时长 + 语调下降（句末）+ 语义完整性（LLM判断）
     - 快速响应：检测到句尾→立即开始LLM推理→不等完整utterance
   - **打断处理（Barge-in）：**
     - 持续运行VAD→即使Agent在播放语音
     - 检测到用户开口→立即停止TTS播放→切换到listening状态
     - 已生成但未播放的音频→丢弃
     - 恢复上下文：被打断的回答记录到对话历史→不丢失
   - **对话状态：** Listening → Processing → Speaking → (被打断→Listening)

5. **工具调用集成：**
   - **挑战：** 工具调用有延迟（API 500ms-2s）→语音对话中不可接受长沉默
   - **解决：**
     - **填充语：** "让我帮你查一下..."→播放自然的过渡语音→同时执行工具
     - **流式结果：** 工具返回部分结果→先说部分→完整后补充
     - **预测性工具调用：** 从partial transcript预判意图→提前发起API调用
   - **Function Calling集成：**
     - LLM输出tool_call → 执行 → 结果注入context → LLM继续生成语音回答
     - 关键：工具调用结果到语音输出的延迟<200ms

6. **延迟优化（达到<500ms端到端）：**
   - **延迟拆解目标：**
     - Audio Frontend: <50ms
     - ASR: <150ms（streaming，不等整句）
     - LLM: <200ms（TTFT，用Prefix Caching + 小模型/Speculative Decoding）
     - TTS: <100ms（流式合成，第一个音素就开始播放）
   - **关键优化：**
     - **Streaming全链路：** 每个组件都是流式→不等前一步完成就开始下一步
     - **Prefix Caching：** system prompt + 对话历史的KV Cache复用→每轮只计算新内容
     - **预计算：** 用户说话时预加载模型/预热TTS
     - **边缘部署：** ASR/VAD在端侧运行→减少网络延迟
     - **WebSocket长连接：** 避免HTTP请求的建连开销

7. **情感感知和表达：**
   - **感知：** 从用户音频提取情感向量→注入LLM prompt→"用户当前情绪：焦虑"
   - **表达：** TTS支持情感控制→"用温和安慰的语气回答"
   - **SSML标签：** `<prosody rate="slow" pitch="low">我理解你的心情</prosody>`
   - **端到端方案：** 音频token天然包含情感→LLM学会"听语气回语气"

8. **关键指标：**
   - **端到端延迟：** P50<400ms, P99<800ms
   - **语音识别准确率：** WER<5%（安静环境）, <10%（噪音环境）
   - **打断响应时间：** 检测到用户开口→停止播放<100ms
   - **对话自然度：** 人工评分1-5→目标>4.0
   - **工具调用成功率：** >95%
   - **并发：** 支持1000+同时在线对话（每个对话一个WebSocket连接）
